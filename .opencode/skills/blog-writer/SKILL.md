## 全局规则

**写作前必须同时读取 `anti-patterns.md` 和 `patterns.md`**，写作全程严格遵守：`anti-patterns.md` 是底线（禁止项），`patterns.md` 是方向（推荐项）。

**资源优先级**：`anti-patterns.md` > `patterns.md` > `styles/` > `samples/` > `categories/`  
当参考资料与禁止项冲突时，**以 `anti-patterns.md` 为准**，忽略资料中违规的用词或结构。

|------|------|
| 描述包含具体主题 + 基本诉求 | 快速通道 |
| 描述模糊，或只说"想写点关于XX的" | 引导通道 |

## 快速通道

### 第一步：推断参数

从用户描述中自动识别：**发布位置**（专栏或自由文章，参考下方映射）、**风格**（默认专业严谨）、**字数**（默认 2000）。

### 第二步：推荐确认

以推荐形式展示推断结果，让用户确认或调整，例如：

> 根据你的描述，我的推荐是：
> - 📁 发布位置：`swd/ai-coding/ai-frontier/`（AI 前沿专栏）
> - 🎨 写作风格：专业严谨
> - 📝 目标字数：约 5000 字
>
> 有需要调整的吗？没有的话直接开始写。

用户确认（或调整某项参数）后再进入写作。

### 第三步：写作

1. 读取 `anti-patterns.md`（必读）
2. 读取 `patterns.md`（必读）
3. 读取 `styles/{风格}.md`
4. 如有 `user-input/` 素材文件，读取
5. 参考 `samples/` 中相关文章（如有）
6. 撰写正文 → 生成 MDX → 执行 `npm run build`

### 第四步：校审

构建成功后自动进入校审，参见下方「校审协议」章节。

## 引导通道

### 第一步：聊出主题（最多2轮）

每次只问一个问题帮助锁定方向。**第2轮后必须给出2-3个方向建议让用户选，不再追问。**

### 第二步：推荐确认参数

格式同快速通道第二步，展示推断的发布位置、风格、字数，让用户确认或调整。

### 第三步：提出大纲

给出3个标题方向，用户选1个（或提出修改）。

### 第四步：写作

同快速通道第三步。

### 第五步：校审

构建成功后自动进入校审，参见下方「校审协议」章节。

## 发布位置：专栏 vs 自由文章

每篇文章有两种发布位置：

**专栏文章**：归属于某个专栏，路径包含专栏子目录。
```
swd/ai-coding/ai-frontier/   ← AI 前沿专栏
swd/ai-coding/claudecode/    ← Claude Code 专栏
swd/xp/tdd/                  ← TDD 专栏
```

**自由文章**：不属于任何专栏，直接落到主题目录。
```
fitness/        ← 运动健康（通常是自由文章）
investment/     ← 投资理财（通常是自由文章）
life/           ← 生活杂谈（通常是自由文章）
swd/agile/      ← 敏捷（也可以是自由文章）
```

推断时优先考虑：内容是否有明显的专栏归属（系列性、专题性）？是则推荐专栏，否则推荐自由文章。

## 输出格式

```markdown
---
title: 文章标题
date: 'YYYY-MM-DD'
tags:
  - 主题标签
  - 用户自定义标签
published: false
brief: 根据全文提炼的200字摘要
---

> 一句话引语或核心观点

正文...
```

规则：
- 正文不出现一级标题 `#`
- 文件名为英文 kebab-case
- `brief` 根据全文内容提炼，200字左右

## 校审协议

**触发时机**：`npm run build` 成功后自动执行。  
**核心原则**：校审结果**只写入报告文件，不修改 MDX 原文**。  
**报告存放**：`.draft/review-{dir}-{slug}.md`（目录不存在则自动创建）。

---

### 一、事实校审

**触发条件**：文章包含以下任一情况时执行；否则跳过，只做风格校审。
- 包含版本号、发布日期、定价、基准测试分数等具体数据
- 包含"X 公司发布 / 宣布 / 表示"等引用性陈述
- tags 含有：`AI前沿`、`大模型`、`技术评测`、`选型指南`、`工具推荐` 等技术类标签

**执行方式**：
1. 从文章中提取所有可验证的客观声明（版本号、日期、数字、引用）
2. **版本货架期检查**：如文章以某产品的基础版本（如 GPT-5）为主角描述，验证截至发布日是否已有更新版本（如 GPT-5.4）成为现役旗舰——如有，标记为 ⚠️ 并建议将主角替换为当前最新版本
3. 用 WebFetch / Tavily 搜索每条声明的最新官方或权威来源
4. **信息源优先级与验证原则**：

   **第一级（最高可信度）：官方来源**
   - 厂商官网发布博客（如 `openai.com/index/`、`anthropic.com/news/`、`deepmind.google/discover/blog/`）
   - 官方 API / 开发者文档（如 `developers.openai.com`、`docs.anthropic.com`、`ai.google.dev`）
   - 官方 GitHub Release Notes 与 Changelog
   - 厂商官方社交媒体（X/Twitter 认证账号、官方 YouTube）

   **第二级：权威科技媒体**
   TechCrunch、VentureBeat、The Verge、Ars Technica、The Decoder、MIT Technology Review、Wired、Reuters、Bloomberg Tech、Fortune、ZDNET

   **第三级：知名 AI 博主 / 社区**
   Simon Willison（`simonwillison.net`）、Andrej Karpathy、LessWrong 精华帖、Hacker News 顶部讨论、DataCamp Blog、Epoch AI Benchmarks

   **冲突处理规则（严格执行）**：
   - 任何来源与第一级冲突 → **以第一级为准**，将冲突来源标记为 ❌ 建议修正，写明"官方数据为 X，与[来源]不符"
   - 多个第二级来源互相矛盾 → 标记为 ⚠️，列出所有版本，说明分歧，提示用户以官方为准
   - 仅有第三级来源 → 可作为线索参考，但不得作为声明的唯一依据，在报告中标注"来源为非权威渠道，建议用官方文档交叉验证"

   **数字 / 日期类声明的强制多源验证**：
   版本号、发布日期、定价、基准测试分数等精确数值，**必须找到至少一个第一级来源确认**；仅有第二/三级来源时，标记 ⚠️ 而非 ✅

5. 按四类打标：

| 标签 | 含义 | 判定标准 |
|------|------|---------|
| ✅ 已验证 | 与权威来源一致 | 找到来源且内容相符 |
| ⚠️ 可能过时 | 信息未被推翻，但找到更新版本 | 来源有效但数据/版本已更新 |
| ❌ 建议修正 | 与权威来源明确不符 | 来源明确指出文章数据/说法有误 |
| 🔍 无法验证 | 主观判断、个人经历、观点陈述 | 本质上不可能有外部来源验证 |

> **边界说明**：  
> - "客观但暂无来源"（如较早的历史数据）→ 归入 ⚠️，说明"未找到权威来源，建议核实"  
> - "个人认为/我觉得"类表达 → 直接归入 🔍，无需搜索  
> - 争议性观点（各方说法不一）→ 归入 ⚠️，列出不同来源并说明分歧

### 二、风格校审

**每篇文章都执行**，不受事实校审触发条件限制。

**执行方式**：
1. 读取 `samples/` 中与本文**同类别**的样例（优先匹配 `domain` 和 `articleType` 元数据，无则取最相近的）
2. 对照以下五个维度评估文章风格：

| 维度 | 评估要点 |
|------|---------|
| 开头方式 | 是否符合个人惯用的开场模式（引语、个人经历、背景铺垫等）|
| 段落节奏 | 段落长短分布是否与样例接近，有无"学术论文感"或"流水账感"|
| 语言温度 | 是否有个人观点的注入，是否过于冷淡或过于口水化 |
| 结尾方式 | 是否以个人感悟/哲学性总结收尾，符合个人风格 |
| Markdown 用法 | 加粗/列表/表格/引用块的使用是否一致、克制 |

3. 给出**具体的调整建议**，每条建议需注明位置、问题、调整方向、参照维度、参考样例

### 四、自动修复分析（校审后立即执行）

校审报告生成后，**立即**对所有 ❌ 项和风格违规项做自动修复分类，**无需等待用户指令**：

#### 分类标准

| 类型 | 判定标准 | 处理方式 |
|------|---------|---------|
| **直接替换型** | 有明确正确值，修改是"字符串替换"级别，不影响周边逻辑 | ✅ 立即自动修复 |
| **风格违规型** | 禁止词出现在标题或 brief 等可预测位置，且替换有标准模式 | ✅ 立即自动修复 |
| **逻辑牵连型** | 修复后会改变文章核心论点，或需要重写多段周边内容 | ⚠️ 标注理由，等待用户决策 |
| **歧义待确认型** | ⚠️ 项，信息有多个版本，或正确值无法从校审中确定 | 🔍 列出已知信息，询问用户 |

**直接替换型的判定规则**（满足以下全部条件）：
- 有具体的"旧值 → 新值"（如 `200K → 1M`、`3月 → 2月`）
- 在文章中出现位置确定，替换不依赖上下文语义
- 修复后，周围句子的逻辑关系不需要重写（例：改个数字，但"远超所有对手"这类依赖该数字的论断需要同步处理，此时升级为逻辑牵连型）

#### 自动修复的执行步骤

1. 遍历所有 ❌ 和风格违规项
2. 对每项做类型判断
3. 批量应用所有"直接替换型"和"风格违规型"修复
4. 执行 `npm run build` 验证构建通过
5. 在报告中追加"自动修复记录"段落，列出已修复项和未自动修复项

#### 自动修复记录格式（追加到报告末尾）

```markdown
---

## 自动修复记录

**修复时间**：{YYYY-MM-DD}  
**构建状态**：✅ 通过 / ❌ 失败

### 已自动修复（N 项）
| 位置 | 修复内容 | 类型 |
|------|---------|------|
| 第 32 行，模型矩阵表 | `200K tokens` → `1M tokens` | 直接替换 |

### 需手动决策（N 项）
| 问题 | 类型 | 等待用户决策的原因 |
|------|------|-----------------|
| Gemini "远超所有对手"论述 | 逻辑牵连 | 修正上下文后该论点不成立，需要重写整段 |
```

> **附：常见"直接替换型"清单**（遇到这些，直接修复不用问）
> - 版本号、发布日期、定价数字、基准分数的点状修改
> - 标题/brief 中的禁止词替换（有明确替代方案时）
> - 脚注编号错位（编号和对应文字不匹配）
> - 链接失效替换

---

> **附：常见"逻辑牵连型"清单**（不自动修复，需列出并等待用户）
> - 某数据修正后，基于该数据的"X 远超 Y"、"X 是唯一能…的"等比较性结论同样失效
> - 结构性错误（如章节顺序、选型建议矩阵的推荐项与正文结论不一致）
> - ⚠️ 项（信息可能过时，但没有确定的正确值）

---

### 五、校审与修复完成后的动作

1. 告知用户报告已保存路径（`.draft/review-{dir}-{slug}.md`）
2. 高亮展示：已自动修复项摘要 + 需手动决策项
3. 等待用户指令：
   - 用户确认所有问题处理完毕 → 将文章 `published` 字段改为 `true`，重新执行 `npm run build`，构建成功后流程结束
   - 用户要求处理逻辑牵连项 → 修改 MDX，重新 `npm run build`，更新报告
