# AI 简报三段式工作流设计方案

- **日期**: 2026-05-09
- **状态**: 待实现
- **项目**: 个人博客 / `skills/ai-briefing`

## 1. 背景

当前 `ai-briefing` skill 同时承接了查询类触发词（如“今天 AI 厂商发布了什么”“厂商动态汇总”）与发布类行为（生成稿件、审核、commit、push）。这会导致一个结构性问题：

1. 用户只是想查询近期动态时，skill 也可能进入成稿乃至发布链路。
2. 查询、成稿、发布三类意图边界不清，后续很难继续增强“查询能力”而不提高误发布风险。
3. 为了降低漏报风险，skill 需要更强的候选发现与重点厂商补检能力；但这种“查询/发现层”不应该直接与“自动发布”绑死。

外部 AIHOT 一类 skill 的优势主要在于“查询/发现层”，而当前 `ai-briefing` 的优势在于“核验/成稿/发布层”。本设计目标是保留现有自动发布能力，同时引入明确的三段式工作流，让 skill 更适合长期演进。

## 2. 目标

1. 将 `ai-briefing` 明确拆成三种模式：查询、成稿、发布。
2. 默认行为改为查询模式，避免新闻查询意图误触发成稿或自动发布。
3. 继续保留自动发布博客能力，但只在明确发布意图下执行。
4. 保留并强化现有事实核验、重点厂商覆盖、去重检查、仓库门禁等能力。
5. 为后续增强“关键词/厂商/时间窗查询”“聚合候选发现层”预留清晰入口。

## 3. 非目标

- 本轮不接入新的外部聚合 API 或 AIHOT 数据服务。
- 本轮不修改博客页面、路由或前端交互。
- 本轮不改写 `source-map.md` 的整体结构，只围绕 workflow 改造 skill。
- 本轮不新增独立查询 skill；仍在 `ai-briefing` 内完成意图分流。
- 本轮不要求真正生成独立 sidecar 文件，只要求在 workflow 中明确内部审核摘要的产出要求。
- 本轮不引入分类参数作为核心输入维度，分类能力保留到后续增强。

## 4. 工作流总览

`ai-briefing` 改造后的工作流分为三段：

### 4.1 查询模式（默认）

用于回答：

- 今天 AI 厂商发布了什么
- OpenAI 最近发了什么
- 最近 3 天 Gemini / Kimi / DeepSeek 有什么更新
- 过去 24 小时 AI 行业有什么大新闻

行为：

- 执行候选发现与轻量核验
- 输出中文查询结果
- 不写文件
- 不 commit
- 不 push

### 4.2 成稿模式

用于回答：

- 帮我整理成今天的 AI 简报
- 起草一篇 AI 简报
- 先生成简报，不要发布

行为：

- 执行查询模式的发现与轻量核验
- 执行正式入稿审核
- 生成完整 Markdown 草稿，但默认仅输出到对话，不写入 `content/ai-briefings/`
- 输出审核摘要
- 不 commit
- 不 push

### 4.3 发布模式

用于回答：

- 生成并发布今天的 AI 简报
- 按默认流程发布今天简报
- 今天的简报可以直接发

行为：

- 执行成稿模式全部步骤
- 再执行阻断式最终审核与仓库门禁
- 全部通过后自动 commit / push

## 5. 意图分流规则

建议在 `skills/ai-briefing/SKILL.md` 中新增 `## 意图分流` 章节，作为整个 skill 的入口规则。

### 5.1 默认行为

默认进入查询模式。

如果用户表达不足以明确说明要“生成稿件”或“发布”，则不能进入后两段。

### 5.2 成稿触发词

以下语义进入成稿模式：

- 写简报
- 起草简报
- 生成 AI 简报
- 整理成简报
- 先生成，不发布

### 5.3 发布触发词

以下语义进入发布模式：

- 发布简报
- 生成并发布
- 直接发布
- 提交并推送
- commit and push

### 5.4 歧义降级

若用户语义存在歧义，例如：

- 今天 AI 圈发生了什么
- 厂商动态汇总
- 最近 OpenAI / Anthropic 有什么

则一律按查询模式处理。

原则：

- 宁可少发布
- 不要误发布

### 5.5 旧兼容入口映射

为兼容当前 skill 的触发语义，建议在 `skills/ai-briefing/SKILL.md` 中新增一张“触发词 -> 模式”映射表：

| 触发词/说法 | 默认模式 | 说明 |
|-------------|----------|------|
| AI 雷达 | 查询 | 只查询，不成稿，不发布 |
| AI 前沿雷达 | 查询 | 只查询，不成稿，不发布 |
| 厂商动态汇总 | 查询 | 只查询，不成稿，不发布 |
| 今天 AI 厂商发布了什么 | 查询 | 只查询，不成稿，不发布 |
| AI 简报 | 查询 | 单独出现时视为查询预览，不直接成稿或发布 |
| 每日简报 | 查询 | 单独出现时视为查询预览，不直接成稿或发布 |
| 写 / 起草 / 生成简报 | 成稿 | 进入成稿模式 |
| 发布 / commit / push / 生成并发布 | 发布 | 进入发布模式 |

并要求同步修改 `skills/ai-briefing/SKILL.md` 顶部 YAML `description`，避免继续表达“这些查询词默认自动发布”。

若仓库里仍保留兼容入口（例如旧的 `ai-frontier-radar` skill、等价兼容壳层，或当前存在的 `.opencode/commands/ai-radar.md`），也必须同步复用同一套意图分流规则与同一份默认厂商覆盖集合，至少保证：

- 旧兼容入口默认只进入查询模式
- 不允许绕过 `ai-briefing` 的意图分流直接触发自动发布
- 验收时要覆盖“新旧入口行为一致”

## 6. 查询模式设计

### 6.1 目标

快速、可信地回答近期动态问题，但不生成博客成品。

### 6.2 输入维度

查询模式支持从自然语言中解析：

- 时间范围
- 厂商范围
- 关键词

### 6.3 流程

1. 解析时间窗口（默认仍为 Asia/Shanghai 下“当前执行/查询时刻向前 24 小时”）。
2. 执行重点厂商优先的候选发现。
3. 对候选做轻量核验：
   - 是否有明确时间证据
   - 是否落在窗口内
   - 是否是确定性动态
   - 是否只是旧闻重复
4. 输出中文结果，并附必要来源。

### 6.4 边界

查询模式不是正式简报，不要求：

- 落盘
- 成文章节结构
- 进入 git / 发布链路

查询模式的输出边界必须明确为三类：

- 已确认动态：有明确时间证据，且满足来源等级要求，可直接给用户
- 待核验线索：只有媒体单源、时间证据不足或仍需补检，只能明确标注为待核验，不得伪装成确定性结论
- 未发现符合窗口的确定性更新：当没有任何确认项时，必须直接说明，而不是强行凑结果

如果重点厂商覆盖未完成、关键源抓取失败或补检未完成，查询模式必须显式说明“查询不完整，不能作为成稿依据”。

查询模式默认不得输出内部覆盖清单、内部审核摘要或任何面向内部流程的检查明细。

查询模式只能基于公开、可核验来源回答；非公开、泄露、NDA、受限访问页面或私下沟通获得的信息一律不得输出给用户，也不得作为待核验线索对外展示。

## 7. 成稿模式设计

### 7.1 目标

生成可审阅的简报草稿，但不自动发布。

### 7.2 流程

1. 执行查询模式的候选发现与轻量核验。
2. 进入正式入稿审核：
   - 时间证据
   - 来源等级
   - 重点厂商覆盖
   - 与前一天简报去重
   - 旧事实仅作背景
3. 生成完整 `.md` 草稿文本，并只在对话中返回。
4. 生成内部审核摘要。
5. 输出给用户草稿内容与审核结果。

### 7.3 阻断条件

以下情况不得进入成稿：

- 时间证据不足
- 重点厂商覆盖检查未完成
- 重大候选仍未确认
- 与前一天重复报道但没有新进展
- 内容质量不足以形成合格简报
- 只有 `[待核验]` 线索，没有可入稿的确定性动态

### 7.4 正式文件边界

由于当前仓库中 `content/ai-briefings/` 下的简报默认视为公开发布内容，成稿模式不得直接写入该目录。

本轮最小实现原则：

- 成稿模式一律不落盘，只输出对话中的完整 Markdown 草稿
- 发布模式通过最终审核后，再一次性写入 `content/ai-briefings/YYYY-MM-DD-ai-briefing.md`
- 若当天正式文件已存在，默认停止；只有用户明确要求覆盖当天简报时才继续

因此，本轮不支持“先保存草稿文件但不发布”的中间态文件落盘行为；若未来需要支持，必须引入专用非公开目录并单独设计。

## 8. 发布模式设计

### 8.1 目标

保留当前自动发布能力，但只在明确发布意图时执行。

### 8.2 流程

发布模式 = 成稿模式 + 最终审核 + 仓库门禁 + 自动发布。

### 8.3 原则

发布不是“成稿后顺手 git 一下”，而是单独的第三段。

只要任一前置检查失败：

- 停止发布
- 不 commit
- 不 push

## 9. 发布前审核门禁

建议在现有基础上明确分成四道门。

### 9.1 门 1：事实审核

检查：

- 所有入稿事件都有来源支撑
- 所有时间都已折算到 Asia/Shanghai
- 都严格落在本期统计窗口
- 重点厂商逐一覆盖完成
- 高价值厂商已做 changelog / release notes 补检
- 没有旧闻当新条目
- 没有和前一天重复报道同一事件

### 9.2 门 2：成稿质量审核

检查：

- 结构完整
- 语言直接、专业、克制
- `## 速览` 每条只描述一个事件
- `## 重点动态` 分层清楚
- `## 为什么值得关注` 不空泛
- 正文在 900~1300 字之间
- 当天新闻多时尽可能接近 1300 字

### 9.3 门 3：内部审核摘要

每次成稿/发布前必须形成内部审核摘要（仅供内部使用，不得写入最终简报公开内容），至少包含：

- 本期时间窗口
- 重点厂商覆盖清单
- 候选条目数
- 剔除项与剔除原因
- 最终入稿项
- 每项确认源
- 是否存在高风险未确认项

本轮内部审核摘要默认只允许存在于：

- 对话输出
- 临时过程数据

本轮明确禁止：

- 落盘为默认审核文件
- stage 到 git
- commit / push

若未来需要引入 sidecar 文件，必须单独开新设计。

### 9.4 门 4：仓库发布门禁

继续保留并强化现有门禁：

- `npm run validate-content`
- `npm run build:ai-data`
- 确认简报索引已更新
- 检查当前分支、upstream、behind/diverged 状态
- 确认没有无关 staged changes
- 只 stage 本次相关文件

任一失败：

- 停止发布
- 输出失败原因
- 不允许 commit / push

## 10. 对现有 `SKILL.md` 的最小改造建议

### 10.1 结构调整

建议把 `skills/ai-briefing/SKILL.md` 的现有主结构调整为：

1. 功能概述
2. 默认行为
3. 意图分流
4. 硬规则
5. 执行流程
   - 1. 确定时间窗口
   - 2. 查询模式
   - 3. 成稿模式
   - 4. 发布模式
6. 输出给用户

### 10.2 表述修正

当前 `skills/ai-briefing/SKILL.md` 中“默认自动发布”的表述需要调整为更准确的版本：

> 本 skill 默认进入查询模式；当用户明确要求生成并发布简报时，执行完整的成稿、审核和自动发布流程。

这样既保留自动发布能力，也避免查询意图误触发发布链路。

### 10.3 `argument-hint` 扩展

建议从：

- `[时间范围] [厂商] [选项]`

扩展为：

- `[时间范围] [厂商] [关键词] [选项]`

模式继续完全由意图分流规则决定，而不是要求用户显式输入结构化“模式”参数。

## 11. 与当前规则的兼容性

本次改造不推翻现有优势规则，以下内容继续保留：

- 时间统一折算 Asia/Shanghai
- 重点厂商逐一覆盖检查
- 与前一天简报去重
- 正文长度控制
- 发布前最终审核
- 仓库安全检查与自动发布门禁

本次改造的重点只是：

- 把查询、成稿、发布三类行为显式分开
- 让自动发布只在明确发布意图下执行

## 12. 验收标准

1. `ai-briefing` 的描述与 workflow 中不再把“查询”与“自动发布”混为一谈。
2. 默认行为被明确定义为查询模式。
3. 旧兼容入口（AI 雷达 / AI 前沿雷达 / 厂商动态汇总 / 今天 AI 厂商发布了什么）默认都只进入查询模式。
4. 单独出现“AI 简报 / 每日简报”时，不得直接进入发布模式。
5. 只有明确生成/起草意图时才进入成稿模式。
6. 只有明确发布意图时才进入发布模式。
7. 自动发布能力仍保留，并要求完整通过审核和仓库门禁。
8. 成稿模式不得写入 `content/ai-briefings/`，且本轮一律不落盘；只有发布模式审核通过后才写正式文件。
9. 查询模式在无确认项时必须返回“未发现符合窗口的确定性更新”；只有 `[待核验]` 时不得升级成稿。
10. 覆盖未完成、关键源失败或重大候选未确认时，不得进入发布，并要输出失败原因。
11. 内部审核摘要被明确要求保留在内部流程中，不进入最终简报公开内容，也不默认落盘/stage/commit。
12. `.opencode/commands/ai-radar.md` 等旧兼容入口必须同步改造，默认进入查询模式，不得绕过意图分流直接进发布模式。
13. 若仍存在其他旧兼容 skill/壳层，其默认行为必须与 `skills/ai-briefing/SKILL.md` 保持一致。
14. 现有时间证据、重点厂商覆盖、去重、最终审核等规则继续生效。

## 13. 后续可选增强（非本轮）

- 增加“查询模式”的固定输出模板。
- 增加真正的内部 sidecar 文件产物（如 `.md` 或 `.json`）。
- 后续评估是否引入外部聚合源作为候选发现层，但严格保持“发现层 ≠ 最终确认层”的边界。
