# Investment Daily Briefing Skill

## 功能概述

面向博客的三段式工作流：追踪投资观察范围内的已确认动态与已官宣未来事件，核验来源与时间证据，并根据用户意图执行查询、成稿或发布。

## 默认行为

| 参数 | 默认值 |
|------|--------|
| 默认模式 | 查询 |
| 默认节奏 | 北京时间早报 |
| 时区 | Asia/Shanghai |
| 正常版字数 | 1200~1500 个中文汉字 |
| 短版字数 | 900~1199 个中文汉字 |
| 正式文件类型 | `.md` |
| 正式输出目录 | `content/investment-briefings/` |
| 发布动作 | 仅在发布模式且审核通过后自动 commit 并 push |

## 意图分流

### 默认规则

默认进入查询模式。只有当用户明确表达“写 / 起草 / 生成简报”时进入成稿模式；只有当用户明确表达“发布 / commit / push / 生成并发布”时进入发布模式。

### 模式定义

- 查询模式：只查询与核验，不落盘，不 commit，不 push。
- 成稿模式：生成完整 Markdown 草稿与内部审核摘要，但不写入公开目录，不 commit，不 push。
- 发布模式：执行查询、成稿、最终审核、仓库门禁，并在全部通过后自动发布。

## 硬规则

### 公开边界

- 不提供买卖建议
- 不荐股
- 不写目标价
- 不写仓位建议
- 不把传闻写进公开稿

### 传闻观察

- `传闻观察` 默认关闭
- 即使临时开启，也只能出现在查询模式输出、内部审核摘要或非公开审核文件中
- 传闻不得进入可发布 Markdown 草稿

### 高共识预期

- `高共识预期` 默认关闭
- 只有用户明确开启时才允许进入 `## 预期观察`
- 必须显式标注为“预期”
- 不得使用确定性语气

### 与上一期去重

- 默认必须对照最近一期已发布的投资每日简报做去重检查
- 不得与前一天 / 上一期简报重复记录同一条已确认动态，除非出现了新的关键增量
- 如果同一主题需要连续跟踪，必须明确写出新的变化点，例如：新增财报数据、新增官宣、口径变化、时间点推进、影响范围扩大
- 仅仅换一种表述、重写标题或重复摘要，不算新条目
- 如无新增关键信息，该条应省略或并入“接下来重点观察”，不要再次作为独立确认动态发布

### 非公开审核文件

成稿与发布模式都必须形成非公开审核文件，路径为：

- `.local/investment-audits/`

该目录必须被 `.gitignore` 忽略，且不得 stage、commit 或 push。

### 正式文件合同

正式文件路径：

- `content/investment-briefings/YYYY-MM-DD-investment-daily-briefing.md`

frontmatter 模板：

```yaml
---
title: "投资每日简报 · YYYY-MM-DD"
date: "YYYY-MM-DD"
brief: "一句话概括本期最重要的投资观察。"
published: true
tags:
  - 投资每日简报
  - 腾讯控股
  - 港股科技
  - Google I/O
---
```

frontmatter `tags` 规则：

- 必带：`投资每日简报`
- 其余补充 2~5 个本期关键标签
- 标签只允许来自三类：本期重点公司、本期主线主题/行业、已官宣且确实是观察重点的事件
- 黑名单企业不得作为标签
- 不使用泛词作为标签，例如：上涨、回调、市场、财报季、波动
- 不使用内部审核说明、未确认日程、排除理由作为标签或正文补充说明

## 执行流程

### 1. 查询模式

输出：

- 已确认动态
- 未来重点观察
- 待核验线索
- 当用户显式开启时，可附加 `高共识预期`

约束：

- 不写文件
- 不 commit
- 不 push
- 不公开传闻区块
- 不重复输出与上一期无新增增量的确认动态

### 2. 成稿模式

输出必须分成两段：

1. 可发布 Markdown 草稿
2. 内部审核摘要

约束：

- 草稿必须使用 `今日确认动态`、`接下来重点观察`、`来源` 固定结构
- 如开启 `高共识预期`，只能进入 `## 预期观察`
- `传闻观察` 只允许出现在内部审核摘要，不得进入草稿正文
- 必须对照上一期已发布简报做逐条去重，避免连续两期出现无新增增量的重复动态
- 内部审核式说明不得进入公开正文，例如“某事件因时间未确认而暂不纳入本期”这类排除理由只能保留在内部审核摘要

### 3. 发布模式

发布模式 = 成稿模式 + 最终审核 + 仓库门禁 + 自动发布。

发布前至少执行：

1. 红线语言审查
2. 来源与时间证据核验
3. 与上一期已发布简报的重复条目检查
4. 黑名单企业排除检查
5. `npm run validate-content`
6. `npm run build:investment-data`
7. 确认 `public/investment-data/briefings/index.json` 已更新
8. git 安全检查

默认发布命令：

```bash
npm run validate-content
npm run build:investment-data
git add content/investment-briefings/YYYY-MM-DD-investment-daily-briefing.md public/investment-data/briefings/index.json public/investment-data/coverage.json
git commit -m "add investment daily briefing for YYYY-MM-DD"
git push
```

## 输出给用户

### 查询模式完成后

- 查询摘要
- 已确认动态
- 接下来重点观察
- 待核验线索（如有）

### 成稿模式完成后

- 草稿正文
- 内部审核摘要

### 发布模式完成后

- 简报文件路径
- 审核结果摘要
- commit hash
- push 结果
