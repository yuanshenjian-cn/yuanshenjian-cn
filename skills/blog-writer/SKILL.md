# Skill: blog-writer

## 启动前必读

1. 读取 `anti-patterns.md`：写作底线，优先级最高，先读先内化。
2. 读取 `rules.md`：确认工程事实、路径、frontmatter、slug 和校验命令。
3. 读取 `patterns.md`：写作方向和可读性标准。

按任务需要再读取：
- `styles/{风格}.md`：确定文章语气和段落节奏。
- `categories/{文章类型}.md`：确定结构建议。
- 1-3 篇 `content/blog/` 真实文章：优先参考同目录、同类型、近期文章。
- `samples/index.md`：仅在真实文章不足时查看样例边界；默认不要直接打开完整样例。
- `user-input/`：仅在用户明确指定素材文件时读取；未指定时不要自行遍历猜测。

**资源优先级**：用户明确要求 > `anti-patterns.md` > `rules.md` > `patterns.md` > `categories/` > `styles/` > 真实文章参考 > `samples/`。

**旧文参考边界**：真实文章可以参考口吻、节奏和读者定位；如果旧文出现分割线、套路总结、陈旧事实或当前禁止项，不要复制。

**样例参考边界**：`samples/` 永远不能覆盖 `categories/`、`patterns.md` 和 `anti-patterns.md`。样例只看开头方式、段落长短、小标题密度和结尾节奏，不参考事实、格式、标题写法或投资/健康/产品建议。

## 工作原则

### 自动推进

用户给出清晰主题或素材时，直接推断目录、slug、文章类型、风格、字数和发布状态，并继续写作。不要为了确认默认值而停下来。

只在以下情况暂停询问：
- 用户要求 `published: true` 但没有明确说“直接发布”或等价表达。
- 目标文件已存在，可能覆盖已有文章。
- 用户要求新增专栏或新增内容目录。
- 用户指定要使用 `user-input/` 素材，但没有给出具体文件名。
- 关键事实无法验证，且会影响文章核心结论。
- 用户明确要求先给选题、大纲或参数确认。

### 成功标准

一篇完成稿必须同时满足：
- 能通过 `npm run validate-post -- --strict-writing "content/blog/.../slug.md"`。
- 位于已存在目录，文件名为英文 kebab-case，slug 全仓唯一。
- frontmatter 包含 `title`、`date`、`tags`、`published`、`brief`。
- 正文无一级标题 `#`，不使用 `.mdx` 或 JSX。
- 开头 3 秒内让读者知道核心判断或阅读收益。
- 每个主要判断有事实、经历、例子或推理支撑。
- 语言像作者在和读者认真说明一件事，而不是模板化综述。
- 校审后确定性问题已直接修复，剩余问题只保留真正需要用户决策的部分。

## 通道判断

| 条件 | 通道 | 行为 |
|------|------|------|
| 主题、素材或目标明确 | 快速通道 | 自动推断参数，直接写作并校审 |
| 只给模糊方向，如“想写点关于 XX” | 引导通道 | 最多问 2 轮；仍不明确时给 2-3 个方向让用户选 |
| 用户明确要润色/改写已有稿 | 修订通道 | 读取原稿，对照本技能规则直接修订并校验 |

## 参数推断

写作前内部确定这些参数，并在最终结果中汇报；除非触发暂停条件，不要等待用户确认。

| 参数 | 规则 |
|------|------|
| 目标目录 | 根据主题匹配 `rules.md` 的真实目录。AI 编程工具优先进入对应 AI 专栏。 |
| 文件名/slug | 英文 kebab-case，基于标题生成；写入前用 `validate-post -- --check-path` 验证。 |
| 文章类型 | 从 `categories/` 中选最贴近的一类；不确定时用“经验分享”或“观点评论”。 |
| 写作风格 | 技术/评测默认“专业严谨”；教程默认“简洁明了”；经历复盘默认“朴实稳重”；故事默认“故事叙述”；`talkshow/` 固定“浮夸搞笑”。 |
| 目标字数 | 用户未指定时默认约 2000 字；数据密集或系列教程可扩展到 3500-5000 字。 |
| 发布状态 | 默认 `published: false`；只有用户明确要求直接发布才写 `true`。 |
| 日期 | 使用当前系统日期 `YYYY-MM-DD`；历史复盘可按用户指定日期。 |

## 快速通道

1. 读取必读文件，并按文章类型读取对应 `styles/`、`categories/` 和真实参考文章。
2. 需要事实依据的主题先收集素材，再写结论；事实不足时补充检索。
3. 对事实敏感主题建立事实素材台账，再决定哪些事实进入正文。
4. 生成标题、slug、目录和 tags。
5. 运行路径预检：
   ```bash
   npm run validate-post -- --check-path "content/blog/{目录}/{slug}.md"
   ```
6. 预检通过后写入 `.md` 文件，默认 `published: false`。
7. 运行严格全文校验：
   ```bash
   npm run validate-post -- --strict-writing "content/blog/{目录}/{slug}.md"
   ```
8. 执行校审协议，直接修复确定性问题。
9. 再运行一次严格全文校验。

## 引导通道

1. 每次只问一个问题，优先确认“读者是谁”和“想让读者带走什么判断”。
2. 最多问 2 轮；仍模糊时给 2-3 个可写方向让用户选。
3. 用户选择后进入快速通道。

## 修订通道

适用于用户给出已有草稿、素材整理稿或要求“润色成博客”。

1. 先判断原稿是否已有 frontmatter；没有则补齐。
2. 保留用户原始观点，不替用户新增立场。
3. 修订重点按顺序处理：结构清晰度、事实准确性、段落节奏、语言人味、格式规范。
4. 如果原稿已有强烈个人表达，优先保留；只修掉口水化、重复和误导性表达。
5. 修订后同样运行 `validate-post` 和校审协议。

## 发布位置

已注册目录和 AI 专栏说明见 `rules.md` 的"真实目录层级"和"4 个 AI 专栏说明"章节，此处不重复维护，以 `rules.md` 为唯一来源。

### AI 专栏路由边界

判断一篇文章是否进入 AI 专栏，参考以下示例：

| 文章主题 | 目标目录 | 理由 |
|---------|---------|------|
| Claude Code 某功能的使用教程 | `swd/ai-coding/claudecode/` | 主体是 Claude Code 工具本身 |
| 用 Claude Code 做代码审查的经验复盘 | `swd/ai-coding/claudecode/` | 主体仍是 Claude Code 工具的具体用法 |
| GPT-5 发布评测 | `swd/ai-coding/ai-frontier/` | 属于大模型发布/评测类 AI 前沿内容 |
| 用 AI 工具之后对软件开发的感悟 | `career/` 或 `swd/agile/` | 主体是职业或开发实践，AI 只是背景 |
| 健身受伤康复经历（提到用 AI 查资料） | `fitness/` | 主体是运动健康，AI 是辅助工具 |

**判断原则**：文章的核心价值是否来自特定 AI 工具的使用或 AI 前沿技术本身？是则进专栏，否则进领域目录。

### 新增专栏（必须暂停确认）

只有用户明确说"新建一个 XX 专栏"或等价表达时，才走新增流程。新增 AI 专栏需同步更新：
- `lib/columns.ts`
- `components/column-icons.tsx`

普通内容目录不等于 AI 专栏，不要误改上述文件。

## 输出格式

```markdown
---
title: "文章标题"
date: 'YYYY-MM-DD'
tags:
  - 标签1
  - 标签2
published: false
brief: >-
  根据全文提炼的 120-200 字摘要。
---

> 一句话提炼文章核心判断。

正文...
```

硬性规则：
- 只输出 `.md` 文件，不产出 `.mdx` 或含 JSX 语法的文件。
- 正文不出现一级标题 `#`。
- 文件名为英文 kebab-case。
- slug 必须全仓唯一，写入前用 `validate-post` 验证。
- `published` 默认 `false`；只有用户明确说直接发布才改为 `true`。
- `tags` 优先复用 `rules.md` 中已有写法。
- `brief` 是给列表页看的摘要，不要写成广告语或空泛结论。

## 写作质量标准

### 像人写

- 用具体场景、个人判断、真实限制或明确对象开头，不用宏大背景开头。
- 允许适度第一人称，尤其是经验、观点和复盘类文章。
- 每 300-500 字至少出现一个具体支撑：例子、经历、数据、对比、命令、场景或反例。
- 不把段落写成“观点句 + 三个抽象名词”的模板。
- 不为了显得客观而平均分配篇幅；真正重要的地方多写，不重要的地方少写。
- 结论要有边界：适合谁、不适合谁、在哪些前提下成立。

### 容易读

- 小标题尽量写成有信息量的判断句，不写空泛名词。
- 段落只承载一个意思；超过 6 句优先拆分。
- 列表用于帮助扫读，不用列表替代论证。
- 表格只在多对象同维度比较时使用。
- 代码、命令、路径放进代码块或行内代码，不混在长句里。
- 结尾给读者一个能带走的判断、做法或问题，不写客套话。

### 事实保守

- AI 前沿、产品价格、版本、政策、投资理财、医疗健康等时效性内容必须使用当前环境可用的联网检索或官方文档核实。
- 优先级：官方来源 > 权威媒体/文档 > 一手社区反馈 > 普通二手转述。
- 无法验证的数据不要写成事实；可改写为“我没有找到可靠来源证明...”或删除。
- 样例文章中的数字、价格、日期、模型名和基准分数不得作为事实来源。

### 事实素材台账

满足事实校审触发条件时，写作前先维护一份内部台账，并在校审报告中保留摘要：

| 字段 | 要求 |
|------|------|
| 声明 | 文章准备使用的具体事实或数据 |
| 来源 | URL、文档名或用户素材文件路径 |
| 来源类型 | 官方 / 权威媒体 / 一手社区 / 用户素材 / 无法验证 |
| 日期 | 发布日期或查询日期 |
| 处理 | 写入正文 / 降级表述 / 删除 / 留给用户决策 |

价格、发布日期、政策、收益率、健康建议、基准分数必须在正文附近标注来源或时间；做不到就不要作为事实写入。

## 校审协议

草稿保存并通过首次 `validate-post` 后自动执行。校审不是只写报告，而是先修稿，再报告。

### 事实校审（按需触发）

满足任一条件即触发：
- 文章包含版本号、发布日期、价格、参数、基准测试分数等具体数据。
- 包含“X 公司发布/宣布/表示”等引用性陈述。
- tags 含有：`AI前沿`、`大模型`、`技术评测`、`选型指南`、`投资理财`、`运动健康`。
- 文章意图是当前选型、购买建议、健康建议或投资判断。

执行方式：
1. 提取会影响结论的客观声明。
2. 用官方来源或可靠来源验证；时效性强的信息必须按当前日期重新确认。
3. 对无法验证或过时的信息直接修正、降级表述或删除。
4. 只有会改变文章立场的问题才留给用户决策。

### 风格校审（每篇必做）

对照五个维度检查并直接修复确定性问题：
- **开头**：是否具体、是否快速给出判断。
- **结构**：是否围绕一个主线展开，有无跑题段落。
- **段落**：是否过长、重复、列表过密。
- **语言**：是否有 AI 味、空话、套话、过度平衡。
- **结尾**：是否给出可带走判断，而不是重复全文。

### 工程校审（每篇必做）

- `validate-post -- --check-path` 在写入前通过。
- `validate-post -- --strict-writing` 在写入后通过。
- 文件路径、slug、frontmatter 与 `rules.md` 一致。
- 没有一级标题、MDX/JSX、错误标签格式或字符串布尔值。

### 报告

报告保存为 `.draft/review-{dir}-{slug}.md`。`{dir}` 用目录路径中的 `/` 替换为 `-`。

```markdown
# 校审报告：{文章标题}

**文章路径**：`content/blog/{目录}/{slug}.md`
**校审时间**：{YYYY-MM-DD}

## 已自动修复

| 位置 | 问题 | 修复方式 |
|------|------|---------|

## 事实校审

| 声明 | 来源 | 查询/发布日期 | 处理结果 |
|------|------|-------------|---------|

## 风格校审

**参照文章**：`content/blog/{真实文章路径}`
**整体判断**：通过 / 已修复后通过 / 仍需用户决策

| 维度 | 判断 | 处理 |
|------|------|------|

## 需用户决策

| 问题 | 需要用户决策的原因 |
|------|------------------|
```

校审完成后：
1. 如果有自动修复，重新运行 `npm run validate-post -- --strict-writing "content/blog/{目录}/{slug}.md"`。
2. 告知用户文章路径、报告路径、发布状态和校验结果。
3. 只展示需要用户决策的问题；没有就明确说无待决策项。

## 构建与预览

- 默认不运行 `npm run build`。
- 用户明确要求最终验证时运行 `npm run build`。
- 完整生产构建是 `npm run build:prod`，会先运行图片优化，可能修改 `public/images/`。
- 本地预览使用 `npm run dev`。
