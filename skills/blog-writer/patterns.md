# 写作推荐建议

> 与 `anti-patterns.md` 配合使用。推荐项是方向，禁止项是底线。

---

## 通用建议（所有文章类型）

### 写作目标

- **给人读，不给模型评分**：文章要有明确读者、具体问题和可带走的判断
- **用具体支撑抽象**：每个核心观点至少配一个场景、例子、数据、命令、对比或反例
- **保留作者的判断感**：可以写"我更倾向于..."，但要说明依据和边界
- **让重点自然不平均**：重要段落展开，不重要的信息压缩；不要为了结构整齐牺牲阅读节奏

### 标题

- **优先写有信息增量的标题**：可以是判断、观点，也可以是清晰的指南/教程型标题；关键是让读者知道这篇文章解决什么问题
  - ✅ 《康复比回本更重要》《初次做技术领导的3个陷阱》
  - ✅ 《Claude Code 模型配置指南》《OpenCode 新手入门：从零开始使用》
  - ❌ 《关于模型选择的思考》《运动受伤后的一些经验》

### 开头

- **第一段即核心论点**：开头让读者知道文章结论，后面是展开和支撑
- **从最具体的点切入**：用一个具体的场景、时间、数据或动作开头，不铺垫背景
- **开篇用引用块点出核心判断**（可选）：`> 一句话核心结论`，让读者 3 秒抓住重点
- **避免说明写作安排**：不要写"本文将从三个方面展开"，直接进入第一个有效信息

### 论证

- **事实先，结论后**：先呈现可观察的事实，再给出判断
- **量化比较**：说"更快"时给出具体快多少；模糊的比较等于没有比较
- **承认局限**：结论适用范围不清楚时，主动说明它在哪些情况下不成立
- **回应读者疑问**：每写完一个判断，想一下读者会问"凭什么"、"对我有什么用"、"有没有例外"
- **少用万能框架**：不要套 SWOT、三段论、总分总，除非它确实最适合这个主题

### 格式

- **列表分组标签用粗体 + 冒号**：`**标签**：` 引出分组，不用破折号
- **粗体只标一个关键点**：每段最多加粗一个词或短语
- **代码块用于命令和路径**：不要夹在正文句子里
- **缩写术语首次出现加中文说明**：`ARR（年度经常性收入）`；API、GPU 等普及型无需标注
- **小标题带信息量**：优先写判断句或问题句，例如 `## 为什么它更适合长上下文任务`，少写 `## 优势分析`

### 结尾

- **给一个可以带走的判断**：结尾给读者可以直接用的结论或行动建议，而非"各有优缺点"
- **结尾向前，不回头**：不重复文章已说过的内容
- **留下边界**：说明这个判断在哪些场景下可能不成立，比空泛升华更可信

### 修稿自检

保存前逐项检查：

- 第一段是否已经给出判断或具体问题？
- 是否存在连续 3 段都没有例子、数据、命令、场景或反例？
- 是否有可以删除而不影响理解的套话？
- 小标题去掉后，文章主线是否仍然清楚？
- 结尾是否新增了判断，而不是复述正文？

## AI 前沿/数据密集文章附加建议

> 以下建议仅适用于：大模型发布评测、技术选型、行业格局分析类文章。其他文章类型不需要满足这些要求。

### 写作前：按文章规模收集素材

| 文章规模 | 素材要求 |
|------|------|
| 短评/发布快讯 | 官方来源 + 1 个可靠补充来源；不强行塞媒体综述 |
| 深度评测/选型指南 | 官方测评数据 + 2–4 个权威媒体或文档来源 + 1–3 个一手社区反馈 |
| 行业格局分析 | 至少覆盖官方信号、权威媒体和社区反馈；必须包含一个审慎或批评视角 |

素材服务于判断，不为凑数量而堆引用。短文章宁可少写，也不要写成模板化资料汇编。

### 数据引用规范

- **数字要带上下文**：不是"提升了 23%"，而是"在指定测试条件下吞吐量提升了 23.5%"
- **来源跟数字在一起**：数据和来源放在同一句话或同一段
- **数据表格标注来源**：用 `<small>*数据：来源，日期。</small>`
- **用比较让数字有感**："1M tokens 相当于约 750 本小说"比单独的数字更有画面感
- **当前性要写清楚**：价格、版本、榜单和政策类信息标注查询日期或发布日期

### 多公司并列顺序

涉及 Anthropic、OpenAI、Google 三家并列时，顺序固定：Anthropic → OpenAI → Google（时间先后决定顺序的情况除外）

### 表格用于多对象比较

三个及以上对象需要对比相同维度时，优先用表格而不是平铺段落
