# AI 软开 | 简单设计的四个价值观

> **[样例元数据]** domain: AI 软开 | articleType: 观点评论

> 简单是一个成年人司空见惯的词，然而，大部分成年人却觉得纯真的孩子才是简单的。

"人活简单点真难啊！"、"系统一不小心就搞复杂了"——这些感慨背后藏着同一个心愿：**保持简单**。

禅宗说人生有三境界：看山是山；看山不是山；看山还是山。对一个软件系统来说，这三境也惊人地相似——简单起步，被复杂需求拉着走，最后再设法回到简单。问题在于，大多数系统停在了第二境，再没回来。

为什么保持简单这么难？我在实践 XP 和 Scrum 的这些年里，逐渐认为根源不在技术，而在价值观。支撑简单设计真正落地的，有四个：

## 专注

有三个场景我见过太多次：没有任何 Tasking 就动手写代码；听到"你这设计也太土了"就开始绞尽脑汁套设计模式；在架构评审上有人说"都什么年代了还不上微服务"就开始拆出个分布式单体。

这三种情况，本质上都是没有专注在当下真实的业务价值上。为未来设计，往往意味着把猜测掺进了代码——既增加了当下的复杂度，又很可能白费力气。

**要始终专注于刚好够用的设计，让系统尽可能跟业务模型保持同步。**

## 克制

克制力是一项需要长期培养的能力。我在做 Code Review 和工作坊时，特别注意开发者嘴里的几个词："我觉得……"、"假如以后……"、"万一有人……"。这些用语背后藏着溢出的想法，很容易演变成过度设计。

曾经有个候选人，一道简单的业务题，硬套了 5 种设计模式，叹为观止。

一个有效的克制方式是培养业务视角：深刻理解业务问题，把需求拆成小颗粒度的任务，每次只聚焦一件事。Think Big，Start Small，Move Fast。

## 懒惰

"恰当的懒惰是一个优秀程序员必备的特质，因为 Ta 总是勤于思考如何才能少做重复的事情。"

这不是在提倡摆烂，而是说懒惰会倒逼人去思考：

- 懒得改太多代码 → 消除重复，隔离变化
- 懒得手工集成和测试 → 自动化 CI/CD
- 懒得重复讲搭建环境 → 写自动化脚本和文档

真正有害的是另一种"勤快"——Ctrl+C、Ctrl+V。这是思想上的懒惰，双手却在重复劳动，是完全相反的方向。

勤于思考，惰于盲行。

## 勇气

XP 和 Scrum 都把勇气列为核心价值观之一，不是偶然。在简单设计上需要的勇气往往是：

- 当系统已经臃肿，有勇气拒绝继续堆砌，提出重构
- 当团队想走捷径，有勇气坚持"这里需要先写测试"
- 当有人觉得"你设计太保守"，有勇气说"刚好够用就够了"

这四个价值观不是孤立的——专注让你看清方向，克制让你不走偏，懒惰让你找到更好的路，勇气让你在压力下还能坚持。

好的系统不是一次设计出来的，是一次次抵抗住诱惑的结果。
