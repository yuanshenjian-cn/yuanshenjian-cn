# Investment Briefing Skill

## 功能概述

面向博客的三段式工作流：追踪投资观察范围内的已确认动态与已官宣未来事件，核验来源与时间证据，并根据用户意图执行查询、成稿或发布。

## 默认行为

| 参数 | 默认值 |
|------|--------|
| 默认模式 | 查询 |
| 默认节奏 | 按期发布 |
| 时区 | Asia/Shanghai |
| 正常版字数 | 1400~1700 个中文汉字 |
| 短版字数 | 1100~1399 个中文汉字 |
| 正式文件类型 | `.md` |
| 正式输出目录 | `content/investment-briefings/` |
| 发布动作 | 仅在发布模式且审核通过后自动 commit 并 push |

## 意图分流

### 默认规则

默认进入查询模式。只有当用户明确表达“写 / 起草 / 生成简报”时进入成稿模式；只有当用户明确表达“发布 / commit / push / 生成并发布”时进入发布模式。

### 模式定义

- 查询模式：只查询与核验，不落盘，不 commit，不 push。
- 成稿模式：生成完整 Markdown 草稿与内部审核摘要，但不写入公开目录，不 commit，不 push。
- 发布模式：执行查询、成稿、最终审核、仓库门禁，并在全部通过后自动发布。

## 硬规则

### 公开边界

- 不提供买卖建议
- 不荐股
- 不写目标价
- 不写仓位建议
- 不把传闻写进公开稿

### 传闻观察

- `传闻观察` 默认关闭
- 即使临时开启，也只能出现在查询模式输出、内部审核摘要或非公开审核文件中
- 传闻不得进入可发布 Markdown 草稿

### 高共识预期

- `高共识预期` 默认关闭
- 只有用户明确开启时才允许进入 `## 预期观察`
- 必须显式标注为“预期”
- 不得使用确定性语气

### 与最近 5 期去重

- 默认必须对照最近 5 期已发布的投资简报做去重检查；若已发布历史不足 5 期，则按已有全部历史检查。
- 不得与最近 5 期简报重复记录同一条已确认动态，除非出现了新的关键增量
- 如果同一主题需要连续跟踪，必须明确写出新的变化点，例如：新增财报数据、新增官宣、口径变化、时间点推进、影响范围扩大
- 仅仅换一种表述、重写标题或重复摘要，不算新条目
- 如无新增关键信息，该条应省略或并入“未来重点观察”，不要再次作为独立确认动态发布

### 非公开审核文件

成稿与发布模式都必须形成非公开审核文件，路径为：

- `.local/investment-audits/`

该目录必须被 `.gitignore` 忽略，且不得 stage、commit 或 push。

### 写作可读性

投资简报面向快速阅读场景，不能写成研究报告长段落。事实、解释和观察要拆开，让读者能通过小标题快速扫到本期重点。

成稿/发布模式下遵守以下写法：

- 开篇导语使用短句和短段落。不要用一个长段同时交代市场背景、公司动态和后续观察。
- 章节正文必须小段落、多段落。单段目标控制在 80~120 个中文汉字内；超过 120 字时，优先按“确认事实、变化点、观察含义”拆段。
- 每个自然段只承载一个意思。不要把多个公司、多个日期、多个指标和观察判断塞进同一段。
- 单句尽量不超过 50 个中文汉字。涉及多个公司、日期或指标时，拆成多句。
- `## 近 24 小时确认动态` 和 `## 未来重点观察` 下优先使用多个 `###` 子标题，每个子标题提炼一个确认动态或观察主题。
- `###` 子标题要写成短判断或短变化点，尽量控制在 20 个中文汉字左右，例如“腾讯财报前看 AI 投入”。不要只写公司名、空泛名词，也不要把完整事实句、日期、多个指标直接搬成标题。
- 每个 `###` 下正文用多个 80~120 字左右的短段描述。先写已确认事实或已官宣节点，再写观察重点；不得把多个不相干主题塞进同一段。
- 列表只适合来源、内部审核摘要、同维度数据枚举或极短速览。公开正文主体不要用长 bullet 列表替代叙述。
- `brief` 应是一句清晰摘要，不写成多事件堆叠的长句。

### 正式文件合同

正式文件路径：

- `content/investment-briefings/YYYY-MM-DD-investment-briefing.md`

frontmatter 模板：

```yaml
---
title: "投资简报 · YYYY-MM-DD"
date: "YYYY-MM-DD"
brief: "一句话概括本期最重要的投资观察。"
published: true
tags:
  - 投资
  - 腾讯控股
  - 港股科技
  - Google I/O
---
```

frontmatter `tags` 规则：

- 必带：`投资简报`
- 其余补充 2~5 个本期关键标签
- 标签只允许来自三类：本期重点公司、本期主线主题/行业、已官宣且确实是观察重点的事件
- 黑名单企业不得作为标签
- 不使用泛词作为标签，例如：上涨、回调、市场、财报季、波动
- 不使用内部审核说明、未确认日程、排除理由作为标签或正文补充说明

### 二次复审

- 成稿模式与发布模式中，在本 skill 完成内部审核摘要后，必须再调用项目级 subagent `investment-briefing-reviewer` 做第二道阻断式复审。
- 传给 reviewer 的材料至少包含：Markdown 草稿、内部审核摘要、本期时间窗口、去重检查结果、黑名单排除结果、最终入稿项与对应来源，以及针对最终入稿项的 `权威性 / 真实性 / 时效性` 自审结论。
- 若 reviewer 判定为“需修改后复审”或“阻断发布”，则本轮只能返回问题与修改要求，不得进入发布门禁、commit 或 push。

## 执行流程

### 1. 查询模式

输出：

- 已确认动态
- 未来重点观察
- 待核验线索
- 当用户显式开启时，可附加 `高共识预期`

约束：

- 不写文件
- 不 commit
- 不 push
- 不公开传闻区块
- 不重复输出与最近 5 期无新增增量的确认动态

### 2. 成稿模式

输出必须分成两段：

1. 可发布 Markdown 草稿
2. 内部审核摘要

约束：

- 草稿必须使用 `## 近 24 小时确认动态`、`## 未来重点观察`、`## 来源` 固定结构
- `## 近 24 小时确认动态` 与 `## 未来重点观察` 下优先使用多个 `###` 子标题组织内容，不要简单堆叠长列表
- 如开启 `高共识预期`，只能进入 `## 预期观察`
- `传闻观察` 只允许出现在内部审核摘要，不得进入草稿正文
- 必须对照最近 5 期已发布简报做逐条去重；若已发布历史不足 5 期，则按已有全部历史检查，避免近 5 期内出现无新增增量的重复动态
- 内部审核式说明不得进入公开正文，例如“某事件因时间未确认而暂不纳入本期”这类排除理由只能保留在内部审核摘要
- 生成前的取舍说明、排序理由、写作意图也不得进入公开正文，例如“本期重点是把已官宣节点重新排序，而不是追逐零碎周末新闻”这类表述只能保留在内部审核摘要
- 正文中凡是写到 `M 月 D 日（周X）`、`YYYY-MM-DD（周X）` 等日期+星期组合，必须显式核对星期是否与日期真实对应；拿不准时宁可只写日期，不要猜星期
- 最终入稿项必须逐条完成 `权威性 / 真实性 / 时效性` 自审：优先原始公告、监管文件、公司 IR、交易所或统计机构原始发布；无法回溯原始源、时间不清或明显滞后的内容不得入稿
- 内部审核摘要完成后，必须调用项目级 subagent `investment-briefing-reviewer` 对草稿和内部审核摘要做第二道阻断式复审
- 第二段“内部审核摘要”末尾必须追加“subagent 复审结论”小节

### 3. 发布模式

发布模式 = 成稿模式 + 最终审核 + 仓库门禁 + 自动发布。

发布前至少执行：

1. 红线语言审查
2. `权威性 / 真实性 / 时效性` 核验
3. 来源与时间证据核验
4. 正文日期与星期一致性核验
5. 与最近 5 期已发布简报的重复条目检查
6. 黑名单企业排除检查
7. 可读性检查：开篇和主体正文没有大段落；章节正文单段目标为 80~120 个中文汉字；主体章节使用有信息量的短 `###` 子标题拆分，而不是依赖长列表
8. 项目级 `investment-briefing-reviewer` 已基于草稿与内部审核摘要完成第二道审核，并明确给出“可进入发布门禁”结论
9. `npm run validate-content`
10. `npm run build:investment-data`
11. 确认 `public/investment-data/briefings/index.json` 已更新
12. git 安全检查

其中第 2 项至少要明确：

- 权威性：是否优先采用原始公告、监管文件、公司 IR、交易所、统计机构或权威媒体
- 真实性：关键数字、日期、任命、财报口径、官宣事项能否回溯到原始源
- 时效性：是否属于本期窗口内的新披露、新官宣或新的关键增量，而非旧闻、转载或回顾

任一子项不过关，都不得进入发布门禁。

每次成稿/发布前必须形成内部审核摘要，至少包含：

- 本期时间窗口
- 去重检查结果
- 黑名单排除结果
- 候选条目数
- 剔除项与剔除原因
- 最终入稿项
- 每项确认源
- 每项最终入稿内容的 `权威性 / 真实性 / 时效性` 自审结论
- 是否存在高风险未确认项
- `investment-briefing-reviewer` 的复审结论

默认发布命令：

```bash
npm run validate-content
npm run build:investment-data
git add content/investment-briefings/YYYY-MM-DD-investment-briefing.md public/investment-data/briefings/index.json public/investment-data/coverage.json
git commit -m "add investment briefing for YYYY-MM-DD"
git push
```

`git push` 后不要立刻手动 purge Cloudflare 缓存，避免 GitHub Pages 还未部署完成时又把旧 HTML 回填进 CDN。默认由 `.github/workflows/deploy.yml` 在 Pages 部署成功后执行定向 purge；只有用户明确要求手动补刷时，才执行 `npm run purge:cloudflare -- --scope investment-briefing --date YYYY-MM-DD`。

## 输出给用户

### 查询模式完成后

- 查询摘要
- 已确认动态
- 未来重点观察
- 待核验线索（如有）

### 成稿模式完成后

- 草稿正文
- 内部审核摘要（含 `investment-briefing-reviewer` 复审结论）

### 发布模式完成后

- 简报文件路径
- 审核结果摘要
- commit hash
- push 结果
