# LLM 之家维护 Skill

## 功能概述

这个 skill 用来长期维护博客专栏 `LLM 之家`。

目标不是写单篇快讯，而是维护一组**按厂商拆分的长期模型档案**，集中记录每家最近 `2~5` 代主力大模型的：

- 官方发布时间
- 官方价格
  - 输入
  - 缓存命中 / prompt cache read / context caching
  - 输出
- 模型特色
- 统一评分维度下的观察依据
- 官方来源

默认厂商名单以 `config/focus-companies.json` 为准。

## 第一原则

### 只追加，不覆写，不删历史

这是这个 skill 的第一硬规则。

第一次建档时：

- 每家优先覆盖最近 `3~5` 代
- 如果官方可靠资料不足，至少保留最近 `2` 代

后续维护时：

- **只能追加新模型记录到原文顶部或表格顶部**
- **不得删除旧代记录**
- **不得把旧代记录改写成只剩最新口径**
- **不得因为官方页面下线就移除旧来源痕迹**

允许做的修正只有：

- 修复明显写错的日期、价格、型号名
- 补充此前缺失的官方来源
- 在原记录上追加“官方现页未保留/官方未公布”等说明

## 第二原则

### 只认官方来源

这个 skill 的审核硬门槛：**信息源必须可靠真实，且优先来自官方。**

允许使用的来源类型：

- 官方博客 / 官方新闻页
- 官方开发文档 / 模型页 / release notes / changelog
- 官方 pricing 页
- 官方 GitHub / Hugging Face / 模型卡

默认**不使用**这些来源填充公开档案：

- 媒体二手汇总
- 社区帖子
- 论坛截图
- 第三方模型对比表
- 无法确认发布时间的转载

如果官方拿不到某个字段，直接写：

- `官方未公布`
- `官方现页未保留`
- `官方页面未直接标注`

不要猜，不要补脑，不要拿二手信息填空。

## 第三原则

### 默认一次只维护一家厂商

这是为了避免一次批量改多篇文章时，口径和证据链一起漂移。

- 用户明确指定一家厂商时，直接处理这家
- 用户明确指定多家厂商时，按厂商拆开逐家处理
- 用户没有指定厂商、只说“更新 LLM 之家”时，默认只做检查和候选更新建议，不直接批量落盘
- 除非用户明确要求批量更新，否则不要在同一轮里同时改多家文章

## 第四原则

### 厂商文章一旦更新，frontmatter 的 `date` 必须改成更新当天

这条规则只适用于**文章 frontmatter 的 `date`**，不适用于模型自己的官方发布日期。

- 只要某家厂商文章发生落盘更新，无论是追加新代还是修旧字段，frontmatter 的 `date` 都要改成这次更新当天的日期
- 表格和正文里的模型 `官方发布日期` 继续保持官方原始日期，不要跟文章 `date` 混写
- 如果这次只是检查、没有落盘，不改文章 `date`
- 日期格式继续使用 `YYYY-MM-DD`

## 默认配置

| 参数 | 默认值 |
|------|--------|
| 默认模式 | 默认一次只维护一家厂商；未指定厂商时只检查，不直接批量落盘 |
| 输出语言 | 中文 |
| 文章目录 | `content/blog/swd/ai-coding/llm-family/` |
| 厂商配置 | `config/focus-companies.json` |
| 来源地图 | `references/source-map.md` |
| 代际下限 | 最近 2 代 |
| 代际目标 | 最近 3~5 代 |

## 配置文件

### `config/focus-companies.json`

这是厂商单一真相源。

每个条目至少包含：

- `slug`：文章文件名前缀
- `name`
- `aliases`
- `enabled`
- `minGenerations`
- `targetGenerations`
- `articlePath`
- `officialSources`

如果后续新增厂商，先改这个配置，再创建对应文章。

### `references/source-map.md`

记录每家厂商推荐优先检查的官方来源入口。

## 适用场景

### 1. 追加某家最新模型

例如：

- `更新 LLM 之家里 OpenAI 的最新模型`
- `把 Claude 最新一代追加进档案`

行为：

1. 查该厂商官方发布页与官方价格页
2. 判断这是不是新的模型代际，而不是单纯平台更新
3. 在原文章顶部追加新模型记录
4. 更新顶部总表
5. 保留旧记录不动

### 2. 校验价格或发布日期

例如：

- `校验 Kimi 档案里的价格`
- `核对 Gemini 2.5 Pro 的发布日期`

行为：

1. 只检查指定字段
2. 如果原记录错误，做最小修正
3. 不重写整篇文章

### 3. 新增一家厂商

例如：

- `把 xAI 也纳入 LLM 之家`
- `新增 Mistral 的模型档案`

行为：

1. 先更新 `config/focus-companies.json`
2. 在 `references/source-map.md` 新增该厂商官方源
3. 新建一篇厂商文章
4. 首版优先覆盖最近 `3~5` 代，至少最近 `2` 代
5. 不新增或维护专栏导读、维护手册一类配套文章

## 更新流程

### 1. 判断是否真的是“新模型”

以下情况通常**不直接算新代际**：

- 只是 endpoint 名变化
- 只是 preview 小版本替换
- 只是平台功能更新
- 只是价格变更，没有新模型

只有满足以下之一，才视作应追加的新记录：

- 官方明确宣布新模型
- 型号名出现新的主版本 / 代际版本
- 上下文、能力、价格结构、模型定位出现明显代际变化

### 2. 收集官方事实

每条新记录至少核实：

- 模型名
- 官方发布日期
- 输入价格
- 缓存命中 / cache read / context caching 价格
- 输出价格
- 1~3 条模型特色
- 3~5 个维度里最重要的可观察依据
- 官方来源 URL

### 2.1 先判断这次是哪种维护动作

开始前先把任务归到下面三类之一：

- **追加新代**：新增一代模型记录与对应判断
- **修旧字段**：只修日期、价格、来源、型号名这类确定性字段
- **新增厂商**：新增配置、来源地图和一篇厂商档案

如果一个请求同时混了多种动作，按这个顺序拆开处理：

1. 先修旧字段
2. 再追加新代
3. 最后才考虑新增厂商

### 2.2 字段检查清单

#### 追加新代

至少核对：

- 模型名
- 官方发布日期
- 输入价格
- 缓存命中 / cache read / context caching 价格
- 输出价格
- 1~3 条模型特色
- 至少 1 条官方来源

#### 修旧字段

只核对被点名的字段，以及会直接受它影响的最小相邻内容：

- 表格单元格
- 脚注或小字说明
- `## 官方来源` 列表

默认不要顺手改这些内容：

- 旧代整段分析
- 评分结论
- 其他没被点名的模型代际

但如果这次发生了实际落盘更新，仍然要同步更新该文章 frontmatter 的 `date`。

#### 新增厂商

至少完成：

- `config/focus-companies.json` 新增条目
- `references/source-map.md` 新增官方入口
- 新建对应厂商文章
- 首版覆盖最近 `2~5` 代，至少最近 `2` 代

### 2.3 价格核对顺序

核对价格时，按这个顺序查：

1. 先看当前官方 pricing 页
2. 如果是旧代价格，再回看对应发布公告、release notes 或模型发布页
3. 如果当前 pricing 页没有旧价，但历史官方发布页保留了价格，就按历史官方口径记录，并注明来源边界
4. 只有当前页和历史官方页都没有，才写 `官方未公布`、`官方现页未保留` 或同义说明

### 3. 选择记录位置

更新现有厂商文章时：

1. 更新顶部总表，把新模型插到第一行
2. 在正文最前面新增这一代的小节
3. 保持旧代正文顺序不动
4. 把文章 frontmatter 的 `date` 改成这次更新当天
5. 不在厂商文章里追加独立的维护说明段落

如果这次是“修旧字段”任务：

1. 只改对应字段
2. 如有必要，只补最小脚注说明
3. 同步把文章 frontmatter 的 `date` 改成这次更新当天
4. 除非旧正文判断直接依赖错误字段，否则不改原判断

### 4. 缺口处理

如果官方资料不完整：

- 允许保留该模型记录
- 但缺失字段必须明确标注 `官方未公布` 或同义说明
- 不得用第三方资料把字段补满

### 5. 新厂商建档

新厂商文章建议结构：

1. 导语：这家最近几代路线怎么变
2. 统一评分维度说明
3. 最近几代总表
4. 每一代模型小节
5. 我的实际判断
6. 官方来源

## 写作与更新边界

- 更新档案时，优先做**最小改动**
- 不顺手重构整篇旧文
- 不因为你现在知道更多，就重写旧代所有判断
- 如果要补结论，只补新代的判断
- 厂商文章不是新闻稿，不要写成“今天宣布了什么”的快讯体

## 校验要求

每次落盘后至少检查：

1. 文章仍是 `.md`
2. frontmatter 合法，且 `date` 已更新为本次落盘日期
3. 顶部总表与正文代际顺序一致
4. 旧代记录仍在
5. 新增事实都有官方来源
6. 无法确认的信息已标成缺口说明

## 输出给用户

### 只检查不落盘

返回：

- 已确认的新模型或修正项
- 本次判定属于哪类维护动作
- 原文是否需要更新
- 若落盘，文章 `date` 应更新为哪一天
- 哪些字段官方仍拿不到

### 落盘更新后

返回：

- 更新了哪篇厂商文章
- 本次执行的是哪类维护动作
- 追加了哪一代模型
- 修正了哪些错误字段
- 文章 frontmatter 的 `date` 已更新为哪一天
- 是否新增了厂商配置或来源地图
- 是否有仍待确认的官方缺口

## 禁止事项

- 禁止删除旧代记录
- 禁止把旧代价格改成现价
- 禁止用媒体或论坛补官方缺口
- 禁止把多家厂商混写进同一篇厂商档案
- 禁止为了凑满 5 代而写入无法确认的旧代信息
