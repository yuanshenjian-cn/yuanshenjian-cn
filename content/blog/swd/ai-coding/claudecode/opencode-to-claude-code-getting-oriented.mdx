## 引言：为什么这不是"换个 CLI"

如果你之前一直用 OpenCode 做 AI 辅助编程，你可能已经习惯了它的模式：在终端里输入自然语言指令，AI 帮你读代码、改文件、跑命令。切到 Claude Code 后，表面上看起来差不多——都是终端里对话、都能编辑文件、都支持 MCP。

但深入使用后你会发现，这两个工具的**设计哲学**截然不同：

| 维度 | OpenCode | Claude Code |
|------|----------|-------------|
| **定位** | 开源社区驱动的 AI Coding Agent | Anthropic 官方深度集成的编码平台 |
| **模型** | 支持 75+ 提供商，模型自由切换 | 以 Claude Sonnet/Opus 为核心，支持有限的第三方 |
| **形态** | 主要是终端 CLI | CLI + VS Code + JetBrains + Desktop App + Web + iOS |
| **扩展** | MCP + AGENTS.md + 自定义 Agent | CLAUDE.md + Skills + Sub-agents + Hooks + MCP + Plugins + Agent Teams |
| **权限** | 基础的工具权限控制 | 细粒度权限系统 + Auto Mode + Sandbox |
| **协作** | 会话分享链接 | Remote Control + Channels + Dispatch + 云端会话 |
| **CI/CD** | 无原生支持 | 原生 GitHub Actions + GitLab CI/CD 集成 |

OpenCode 的核心优势是**开放性**——你可以选任何模型、任何提供商，代码完全透明。而 Claude Code 的核心优势是**深度集成**——它把 AI 编程从"一个终端工具"变成了"一个跨平台的开发协作平台"。

这篇文章不会手把手教你安装和使用（那是后续文章的事），而是帮你建立正确的**心智模型**，让你知道：

1. 两者的核心差异在哪里
2. Claude Code 的全景能力地图长什么样
3. 迁移时该关注什么、该忽略什么
4. 这个系列 8 篇文章如何串起来

## 心智模型：从"开源瑞士军刀"到"深度集成平台"

### OpenCode 的心智模型

OpenCode 像一把**开源瑞士军刀**：

```
你 → 选择模型提供商 → 终端 CLI → AI 读/写/执行 → 输出
```

它的核心价值主张是"自由"：自由选择模型、自由配置 API Key、自由定制 Agent。你完全控制每一个环节，但也需要自己搭建和维护整个工具链。

### Claude Code 的心智模型

Claude Code 更像一个**深度集成的开发平台**：

```
你 → 选择入口（CLI/IDE/Desktop/Web/Mobile） → Claude 代理循环 → 内置工具 + 扩展层 → 输出
                                                    ↑
                                          CLAUDE.md / Skills / Hooks / MCP / Plugins
```

关键的思维转变：

1. **从"选模型"到"选入口"**：OpenCode 的核心决策是选哪个模型提供商；Claude Code 的核心决策是选哪个入口和工作模式
2. **从"手动配置一切"到"渐进式扩展"**：Claude Code 开箱即用就很强，扩展是锦上添花而非必需
3. **从"单会话"到"跨端跨会话"**：你可以在终端开始，手机上继续，桌面应用里审查
4. **从"工具"到"平台"**：Claude Code 不只是一个 CLI，它是一套包含 CI/CD、定时任务、多代理协作的完整平台

## Claude Code 产品地图

### 五种主要入口

Claude Code 提供五种主要的使用入口，每种适合不同的场景：

| 入口 | 形态 | 核心优势 | 适合场景 |
|------|------|---------|---------|
| **Terminal CLI** | 终端命令行 | 功能最完整、可组合、支持管道 | 日常开发、脚本自动化、CI/CD |
| **VS Code 扩展** | IDE 内嵌面板 | 内联 diff、@-提及、计划审查 | 需要可视化 diff 的开发 |
| **JetBrains 插件** | IDE 内嵌面板 | 交互式 diff、选择上下文共享 | Java/Kotlin/Python 开发者 |
| **Desktop App** | 独立桌面应用 | 可视化 diff、并行会话、Dispatch、定时任务 | 多任务管理、云端会话 |
| **Web** | 浏览器 | 无需本地安装、长时间运行任务 | 远程仓库、移动端、排队运行 |

### 超越入口：跨端协作能力

除了五种入口本身，Claude Code 还提供了一组**跨端协作**能力，这是 OpenCode 完全没有的维度：

| 能力 | 作用 | 使用场景 |
|------|------|---------|
| **Remote Control** | 从手机/平板/任何浏览器继续本地 CLI 会话 | 离开工位但想继续监控任务 |
| **Dispatch** | 从手机发送任务，Desktop App 自动创建会话 | 通勤时快速派发任务 |
| **`/teleport`** | 将 Web 会话拉入本地终端 | 云端跑完后想在本地继续 |
| **`/desktop`** | 将终端会话交给 Desktop App | 需要可视化审查 diff |
| **Channels** | 从 Telegram/Discord/Webhook 推送事件到会话 | 外部事件触发处理 |
| **Slack 集成** | 在 Slack 中 @Claude 触发任务 | 团队协作、错误报告转 PR |
| **iOS App** | Claude iOS 应用中使用 Code 功能 | 移动端启动长时间任务 |

> **迁移者注意**：如果你只是把 Claude Code 当作一个终端 CLI 来用（就像用 OpenCode 一样），你只发挥了它不到 30% 的能力。

## 扩展能力：从 OpenCode 的简单到 Claude Code 的丰富

这是迁移时变化最大的部分。OpenCode 的扩展能力主要就三个：AGENTS.md、MCP、自定义 Agent。而 Claude Code 有一整套分层扩展体系：

### 扩展层全景

```
┌──────────────────────────────────────────────────┐
│                  打包与分发层                      │
│           Plugins / Marketplaces                  │
├──────────────────────────────────────────────────┤
│                  外部连接层                        │
│           MCP（数据库、Slack、Chrome…）             │
├──────────────────────────────────────────────────┤
│                  事件驱动层                        │
│           Hooks（26 个生命周期事件）                │
├──────────────────────────────────────────────────┤
│                  隔离执行层                        │
│     Sub-agents / Agent Teams（独立上下文）          │
├──────────────────────────────────────────────────┤
│                  按需加载层                        │
│           Skills（知识、工作流、命令）               │
├──────────────────────────────────────────────────┤
│                  始终加载层                        │
│     CLAUDE.md / .claude/rules/（项目约定）          │
└──────────────────────────────────────────────────┘
```

### 与 OpenCode 的扩展能力对照

| OpenCode 概念 | Claude Code 对应 | 迁移说明 |
|---------------|-----------------|---------|
| `AGENTS.md` | `CLAUDE.md` + `.claude/rules/` | Claude Code 兼容读取 AGENTS.md，推荐在 CLAUDE.md 中用 `@AGENTS.md` 导入 |
| MCP 服务器配置 | MCP（配置方式不同） | 需要重新配置，语法有差异 |
| 自定义 Agent（Sub Agent） | Sub-agents + Agent Teams | 能力大幅增强，支持隔离上下文和多会话协作 |
| 无对应 | Skills | **全新能力**：可重用知识和可调用工作流 |
| 无对应 | Hooks | **全新能力**：26 个生命周期事件的自动化 |
| 无对应 | Plugins / Marketplaces | **全新能力**：打包和分发扩展 |
| 会话分享 | Remote Control / Channels / Web | 从"分享链接"升级到"跨端实时协作" |
| 无对应 | Auto Memory | **全新能力**：自动记忆系统 |

> **关键认知**：Claude Code 的 Hooks 和 Skills 是迁移后最大的能力增量。Hooks 让你在 AI 操作的生命周期中插入自动化脚本；Skills 让你把重复的工作流封装成可调用的命令。这两个能力在 OpenCode 中完全没有对应物。

| 概念 | OpenCode 叫法 | Claude Code 叫法 | 说明 |

## 迁移清单：切换后第一周该做的事

### 第一天：基础迁移

- [ ] **安装 Claude Code**：推荐使用 Native Install（`curl -fsSL https://claude.ai/install.sh | bash`）
- [ ] **登录认证**：使用 claude.ai 账户或 API Key
- [ ] **迁移 AGENTS.md**：在项目根目录创建 `CLAUDE.md`，用 `@AGENTS.md` 导入现有规则

```markdown
# CLAUDE.md 示例
# 导入现有 AGENTS.md（避免双份维护）
@AGENTS.md

# 添加 Claude Code 特有的指令
## 构建与测试
- 运行 `npm run build` 验证构建
- 运行 `npm run test` 验证测试
```

### 第二天：权限配置

- [ ] **了解权限模式**：先用默认模式（逐一确认），熟悉后可切换到 Auto Mode
- [ ] **配置权限规则**：在 `.claude/settings.json` 中设置项目级权限

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git *)",
      "Read",
      "WebFetch"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)"
    ]
  }
}
```

### 第三天：MCP 迁移

- [ ] **迁移 MCP 配置**：Claude Code 的 MCP 配置在 `.claude/settings.json` 或 `~/.claude.json` 中
- [ ] **验证连接**：使用 `/mcp` 命令检查所有 MCP 服务器的连接状态

### 第四天：探索新能力

- [ ] **尝试内置 Skills**：运行 `/simplify`、`/batch`、`/debug` 等捆绑命令
- [ ] **体验 Auto Memory**：让 Claude 工作一段时间，观察它自动记住的内容
- [ ] **尝试 Plan Mode**：在复杂任务前用 Plan Mode 先规划再执行

### 第五天：高级配置

- [ ] **创建自定义 Skill**：把你最常用的工作流封装成 `/命令`
- [ ] **配置 Hooks**：为常见操作添加自动化（如保存后自动格式化）
- [ ] **设置 `.claude/rules/`**：将特定于文件类型的规则拆分出来

## 你可能失去的（以及为什么值得）

坦诚地说，从 OpenCode 迁移到 Claude Code，你确实会失去一些东西：

| 失去的 | 得到的 |
|--------|--------|
| 75+ 模型提供商自由选择 | Claude 系列模型的深度优化（代理循环专为 Claude 设计） |
| 完全开源、代码透明 | 更成熟的产品体验和更快的迭代速度 |
| 免费模型（Kimi K2.5 等） | 需要付费订阅（Pro/Max/Team） |
| 社区驱动的扩展生态 | 官方维护的 Plugin Marketplace |
| 简单直接的配置 | 更复杂但更强大的扩展体系 |

**什么时候不该迁移**：
- 你强烈需要使用非 Claude 的模型（如 GPT-4o、Gemini）
- 你的工作环境完全无法联网
- 你需要完全开源透明的工具链
- 你的团队没有预算支付 Claude 订阅

**什么时候值得迁移**：
- 你需要跨端、跨会话的协作能力
- 你想要自动化 CI/CD 集成
- 你需要细粒度的权限控制
- 你想利用 Hooks 和 Skills 构建自动化工作流
- 你的团队需要统一的 AI 编码平台

## 小结

从 OpenCode 到 Claude Code 的迁移，本质上是从一个"开源灵活、模型自由"的终端工具，切换到一个"深度集成、跨端协作"的 AI 编码平台。

你需要调整的不只是配置文件，更是**心智模型**：

1. Claude Code 不只是一个 CLI，它是一个跨五种入口的平台
2. 扩展能力从 OpenCode 的三层（AGENTS.md + MCP + Agent）变成了七层（CLAUDE.md + Rules + Skills + Sub-agents + Hooks + MCP + Plugins）
3. 跨端协作（Remote Control、Channels、Dispatch）是 Claude Code 独有的维度
4. Hooks 和 Skills 是迁移后最大的能力增量
