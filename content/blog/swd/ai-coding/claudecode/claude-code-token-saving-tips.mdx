> 一句话结论：Claude Code 省 Token，靠的不是手快去 `/clear`，而是把上下文管理好，把 Token 花在真正有用的信息上。

## 先说一个底层事实

Claude Code 每次发起请求时，都会重新处理当前输入。输入通常包括三部分：

- 固定内容：系统指令、工具定义、`CLAUDE.md`
- 对话历史：当前会话前面所有轮次
- 新消息：新输入的内容

所以真正决定成本的，不是新输入的这一句话，而是前面那一大段上下文是不是还值得继续带着。

## 10 条低成本使用原则

### 1. 默认策略不是机械地 `/clear`，而是按任务边界管理上下文

要不要清上下文，得先看任务有没有变， 而不是固定 "聊几轮就清一次"。

官方的建议：**切换到不相关任务时，用 `/clear` 重新开始**。但这不等于每做完一步、每问几轮就应该清空一次。

如果你还在做同一件事，继续当前会话通常更合理，因为：

- 已加载的项目规则和上下文还在发挥作用
- Claude Code 会自动使用 prompt caching
- 自动压缩会在上下文接近上限时帮助你保留摘要

这里简单介绍一下缓存到底是怎么“失效”的。

命中的缓存内容按 **cache read** 计费，官方价格是基础输入价格的 **0.1x**。所以对日常使用 Claude Code 来说，更实用的理解不是“这个会话聊了几轮”，而是：**前面那段稳定上下文有没有在短时间内继续被复用。**

而缓存常见的失效方式主要有两种：

- **超过默认 TTL 没再用到**：1小时的时间窗口，API 用户 5 分钟，超过这段时间后，之前的缓存前缀就可能不再命中。
- **可复用前缀被改坏了**：官方文档明确提到，像工具配置变化、提示前缀变化，或者是否包含图片这类变化，都会打断缓存命中。

如果想尽量保持缓存有效，最实用朴素的做法是：

- 同一个任务尽量连续做，不要做两步就切去别的话题
- 不要在主会话里频繁引入无关的大段内容
- 不要无必要地来回改模型、工具配置或上下文结构

以下是一些判断参考依据：

- 任务没变
- 距离上一条消息不久
- 代码和讨论过的方案仍然有用

### 2. 三种情况更适合开新会话或直接 `/clear`

有些时候，继续聊下去就是在给噪音付费了。下面这几种情况，我基本都会直接重开：

- 任务切换了：比如刚修完登录模块，又开始做支付模块
- 当前上下文已经塞进太多无关探索内容
- 当前会话已经离题太远，继续压缩和续聊都不如重新开始直接

一句话判断就是：**任务已经换了，或者上下文已经脏了，就可以开始新的。**

### 3. 长内容给文件路径，不要直接粘进对话

最便宜的 Token，是还没来得进入上下文的 Token。

尽量不要把几千行日志、超长报错、整段文档直接贴进对话里。更好的做法是：

- 提供文件路径
- 提供错误命令
- 提供搜索范围

这样 Claude Code 会自己用工具去找关键片段，而不是把整份内容都吞进上下文。后者通常又贵又乱。

### 4. 简单任务降低 `effort`，复杂任务先 Plan Mode 再执行

现在推荐用 `effort` 来控制成本和深度，而不是只盯着“thinking token”。

实际用下来也确实是这样：简单任务开太高的 `effort`，多数时候是在浪费；复杂任务如果一开始没想清楚，后面反复返工反而更贵。

建议是：

- 小改动、改文案、改样式、局部重命名：用更低的 `effort`
- 跨文件重构、架构调整、复杂 Bug：先做计划，再实施

常用入口：

- `/effort low|medium|high|max`
- 在 `/model` 里调整 effort
- 复杂任务先进入 Plan Mode

尽可能减少返工可以省下大量的Token。

### 5. 1M 上下文窗口能用，但不要默认放任它膨胀

1M 上下文当然是个很强的能力，官方也已经确认 Opus 4.6 和 Sonnet 4.6 支持它。但我的建议一直是：**能用，不代表应该无脑放开。**

原因也不复杂。上下文越长，管理成本越高，噪音越容易堆积，后面你自己都未必分得清哪些信息还值得继续带着。

更稳一点的做法是：

- 保留大窗口能力
- 同时设置保守的自动压缩阈值

例如：

```json
{
  "env": {
    "CLAUDE_CODE_AUTO_COMPACT_WINDOW": "200000"
  }
}
```

如果你明确不想启用 1M 上下文，也可以直接关闭：

```json
{
  "env": {
    "CLAUDE_CODE_DISABLE_1M_CONTEXT": "1"
  }
}
```

### 6. 日常开发优先 Sonnet，复杂推理再上 Opus

这个判断其实和选机器配置很像，不是永远上最贵的就最好。

大多数日常编码任务，Sonnet 已经够用；Opus 更适合：

- 复杂架构设计
- 长链路推理
- 大范围跨文件重构

所以真正该问的，往往不是“怎么让 Opus 少花一点”，而是“这件事到底值不值得上 Opus”。

### 7. 轻量侧任务优先交给低成本子智能体

如果你只是想把一些边角活分出去，我会更建议用子智能体，而不是在主会话里来回切模型。

官方文档也更支持这种思路：把高输出、低复杂度的侧任务交给更便宜的子智能体。

典型场景：

- 跑测试
- 查文档
- 处理日志
- 做只读型代码探索

官方也明确提到，简单的 subagent 任务可以直接指定 `model: haiku`。这很适合做“把杂音隔离出去”的工作，把主会话留给真正需要上下文连续性的任务。

### 8. 精简 `CLAUDE.md`，谨慎增加 Skills 和 MCP

`CLAUDE.md` 这件事，我现在的理解很直接：它不是知识库，它是操作手册。

因为它会在会话启动时加载进上下文，所以它应该短、小、稳定，只保留长期有效的规则。适合长期放在里面的是：

- 项目命令
- 编码规范
- 目录约定
- 不能碰的边界

不适合长期塞进去的是：

- 很长的操作手册
- 偶尔才会用到的流程文档
- 大段背景材料

按最新官方文档，Claude Code 的 MCP tool search 默认开启：启动时通常只有工具名进入上下文，完整 schema 会延迟到真正需要时再加载。所以如果讨论的是 **Claude Code CLI 本身**，那么“装几个 MCP 就立刻把上下文撑爆”这个说法，现在已经不准确了。

不过要具体看：

- 如果是 **非 Anthropic 官方的 `ANTHROPIC_BASE_URL`**，官方文档明确说明：tool search 默认会被关闭，除非显式设置 `ENABLE_TOOL_SEARCH=true`
- 如果是 **更早版本**，或者别的 Claude 客户端，也没有这个默认行为。

如果你想手动控制 ToolSearch，建议直接写进 Claude 的配置文件。官方文档也明确写了：这些环境变量都可以放进 `settings.json` 的 `env` 字段。

常见位置有三个：

- `~/.claude/settings.json`：用户级，作用于你所有项目
- `.claude/settings.json`：项目级，可提交到仓库里给团队共享
- `.claude/settings.local.json`：本地项目级，只对你自己生效

最常用的几种写法如下：

```json
{
  "env": {
    "ENABLE_TOOL_SEARCH": "true"
  }
}
```

上面这段表示：**强制开启 ToolSearch**，所有 MCP 工具都按需延迟加载。

```json
{
  "env": {
    "ENABLE_TOOL_SEARCH": "auto"
  }
}
```

"auto"表示进入**阈值模式**。如果工具 schema 总量不大，就先加载；超过阈值再延迟加载。`auto` 默认按 **10% 上下文窗口**判断。

```json
{
  "env": {
    "ENABLE_TOOL_SEARCH": "auto:5"
  }
}
```

"auto:5"表示把阈值调得更激进一些，按 **5%** 来判断。

```json
{
  "env": {
    "ENABLE_TOOL_SEARCH": "false"
  }
}
```

**关闭 ToolSearch**，启动时完整加载 MCP 工具 schema 的方式，上下文会大很多，不建议关闭。

这里有两个细节很值得记住：

- `auto` **不是默认值**。默认值是按需延迟加载，跟 `true` 一样；`auto` 是阈值模式，按 10% 上下文窗口来判断是否先加载 schema
- ToolSearch 需要支持 `tool_reference` 的模型，官方文档明确写了：**Sonnet 4+ / Opus 4+ 支持，Haiku 不支持**

可以在 Claude Code 里运行 `/context`。最新版本的 Claude Code 不配置时默认开启 ToolSearch。

但结论还是：不用的 Skills、Rules、MCP Tools 应该及时清理，因为它们会增加选择噪音、搜索范围和管理复杂度。

### 9. 命令行优先，MCP 其次

如果一件事命令行就能解决，我一般不会先上 MCP。

最新官方对此也给了明确建议。虽然 MCP 现在支持延迟加载，启动成本比早期版本低很多，但 CLI 通常还是更轻、更直接。典型例子就是：

- GitHub 操作优先使用 `gh`，比如比较流行的 gitub skill，它的 SKILL.md 很简短，主要描述的是如何使用 gh 来操作 GitHub。
- 企业微信CLI、飞书CLI这些也建议让Agent直接使用（LLM 如果没有学习过类似知识，可通过Skill来指导）。
- 文件搜索优先 `grep` / `glob`
- 常规脚本优先 shell 命令

原则不是“别用 MCP”，而是“只有当 MCP 明显更顺手、更强时再用”。

### 10. 用权限控制减少无关扫描

很多 Token 的浪费，其实不是花在回答上，而是花在“到处乱翻文件”上。

可以在 `.claude/settings.json` 里通过 `permissions.deny` 提前排除高噪音目录，例如：

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./secrets/**)",
      "Read(./node_modules/**)",
    ]
  }
}
```

这样做的直接收益很实际：

- 降低误读无关内容的概率
- 缩短模型在大仓库里的搜索路径

当然 Coding Agent 默认是不会去读 .gitignore 里 ignore 掉的文件，如果你是Git仓库，这些文件会被自动忽略。

## 两个补充判断

### 子智能体适合做上下文隔离

一般建议把子智能体当成“隔离舱”来用。它不一定最省单次成本，但特别适合把“测试输出、代码审查、文档搜索”这类长输出隔离出去，最后只把摘要带回主会话。对主会话来说，这通常更省。

### 多智能体并发不等于更省

多智能体并发这件事，很容易让人误以为“更先进也更省”，其实不是。它适合加速复杂任务，但并发实例越多，总体 Token 消耗通常也越高。官方文档提到：在 plan mode 下，agent teams 的 Token 消耗大约可能是标准会话的 7 倍量级。它优化的是速度，不是成本。

## 我会怎么用

如果把上面的内容压缩成我自己的日常习惯，大概如下 7 条：

1. 一个会话只做一件事（相关的任务）
2. 任务没变、当前上下文仍然相关时，继续当前会话
3. 超长日志和大文档只给路径，不整段粘贴
4. 简单任务用轻模式，复杂任务先 plan 再执行
5. 设置自动压缩阈值，避免长会话无限膨胀
6. 定期清理过长的 `CLAUDE.md`、无用 Skills 和不用的 MCP
7. 优先使用命令行工具，必要时通过简单的 Skill 来包装CLI的使用。


## 总结

Claude Code 的成本控制，说到底不是“少说几句话”，而是两件事：

- 让已经付过钱的上下文尽量继续命中缓存
- 不让无关内容进入上下文，避免为噪音反复付费

所以，真正高效的习惯不是“总是开新会话”，也不是“死守一个超长会话不放”，而是根据任务边界、上下文洁净度和当前工作阶段来判断。

当你开始用这个思路管理会话时，Token 消耗通常会比单纯依赖 `/clear` 可控得多，而且回答质量也更稳。
