## 为什么需要"工作流"思维

很多开发者初次使用 AI 编程工具时，把它当做**搜索引擎**——"怎么写一个排序算法？"。但 Claude Code 是**代理式编码环境**：它能读文件、跑命令、改代码、看结果，然后基于结果继续迭代。

这意味着你不是在"问问题"，而是在**指挥一个初级工程师**。你描述目标、提供上下文、给出验证标准，Claude 去执行。执行过程中你可以随时纠偏、中断、回退。

本文覆盖六个最高频的日常场景，每个场景都给出：

- **你怎么提问**（提示词示例）
- **Claude 会做什么**（典型行为链）
- **常见陷阱与应对**

## 场景一：快速理解陌生代码库

### 适用时机

- 刚加入一个新项目
- 接手一个不熟悉的模块
- 在大型 monorepo 中定位功能代码

### 推荐步骤

**第一步：获取全局概览**

```text
给我这个代码库的整体概览：
- 目录结构和主要模块
- 核心技术栈
- 数据流和架构模式
```

Claude 会读取 `package.json`、`README`、关键目录结构，给你一张"地图"。

**第二步：深入特定领域**

```text
解释一下认证模块是怎么工作的。
从前端登录表单开始，追踪到数据库的完整请求流程。
```

**第三步：定位关联代码**

```text
找到所有处理支付回调的文件，
说明它们之间的调用关系。
```

### 提效技巧

| 技巧 | 说明 |
|------|------|
| **从广到窄** | 先问"整体架构"，再问"这个模块怎么工作" |
| **使用领域语言** | 用项目中的术语提问，而不是泛泛的"这个功能" |
| **`@` 引用文件** | `解释一下 @src/auth/middleware.ts 的逻辑` 比 "找到认证中间件并解释" 更精确 |
| **安装代码智能插件** | 运行 `/plugin` 搜索你语言的代码智能插件，让 Claude 获得"转到定义"和"查找引用"能力 |

### 常见陷阱

> **陷阱**：问 Claude "把整个代码库都看一遍"——这会读取大量文件，快速填满上下文窗口。
>
> **应对**：限定范围。"只看 `src/auth/` 目录"比"看整个代码库"高效得多。或者使用 subagent 做探索，避免污染主会话上下文。

## 场景二：从错误描述到修复完成

### 适用时机

- 收到一个 Bug 报告
- 构建或测试报错
- 生产环境出现异常

### 推荐步骤

**第一步：分享错误信息**

```text
运行 npm test 时出现了以下错误：
[粘贴错误日志]

帮我找到根本原因。
```

> **关键原则**：告诉 Claude **重现问题的命令**，而不只是错误消息。Claude 可以自己跑命令获取完整堆栈。

**第二步：让 Claude 诊断**

Claude 典型的行为链：
1. 读取相关源文件
2. 分析错误堆栈中涉及的调用链
3. 定位到具体的问题代码
4. 提出修复方案

**第三步：应用修复并验证**

```text
用你建议的方案修复这个问题。
修完后运行 npm test 确认测试通过。
```

### 提效技巧

| 技巧 | 说明 |
|------|------|
| **给重现步骤** | "登录后点击设置页面会报错" 比 "有个 Bug" 有用得多 |
| **说明是否间歇性** | "这个错误偶尔出现"会影响 Claude 的诊断策略 |
| **要求解决根因** | "解决根本原因，不要抑制错误" 可以防止 Claude 只加 try-catch |
| **粘贴截图** | 直接拖放错误截图到终端，比手动描述更准确 |

### 常见陷阱

> **陷阱**：Claude 修了一个错误，但引入了新的回归。
>
> **应对**：始终要求 Claude 运行完整测试套件，而不只是验证改动的那个测试。

## 场景三：先规划再重构

### 适用时机

- 更新遗留代码到现代模式
- 大范围的 API 变更
- 提取公共逻辑、拆分大文件

### 推荐工作流：Plan Mode 四阶段

**阶段一：探索（Plan Mode）**

按 `Shift+Tab` 切换到 Plan Mode，或启动时使用：

```bash
claude --permission-mode plan
```

```text
读取 src/utils/ 目录，分析其中哪些函数使用了
已废弃的 API。列出每个文件的问题和严重程度。
```

在 Plan Mode 中 Claude 只读取和分析，不做任何修改。

**阶段二：规划**

```text
我想把这些工具函数从 CommonJS 迁移到 ES Modules。
哪些文件需要改？改动的顺序是什么？有哪些风险？
给我一个详细的实施计划。
```

> **进阶技巧**：按 `Ctrl+G` 可以在外部编辑器中打开输入框，你可以在编辑器中编写和修改长 prompt 后发送给 Claude。

**阶段三：实现（Normal Mode）**

切回 Normal Mode（`Shift+Tab` 循环切换），让 Claude 执行计划：

```text
按照你的计划开始重构。
每改完一个文件就运行 npm test 确认没有回归。
```

**阶段四：提交**

```text
用描述性的 commit 信息提交更改。
```

### 何时跳过规划

| 场景 | 需要规划？ |
|------|----------|
| 修改多个文件的架构变更 | ✅ 强烈建议 |
| 不确定最佳方案 | ✅ 建议 |
| 一句话能描述清楚的 diff | ❌ 直接执行 |
| 修复拼写错误、重命名变量 | ❌ 直接执行 |

## 场景四：补测试、补边界、跑回归

### 适用时机

- 给未覆盖的代码补测试
- 发现边界条件缺失
- 新功能开发完成后的验证

### 推荐步骤

**第一步：识别未测试代码**

```text
找出 NotificationsService 中没有被测试覆盖的函数。
```

**第二步：生成测试脚手架**

```text
为通知服务添加测试。
遵循项目现有的测试风格和框架。
```

Claude 会检查项目中已有的测试文件，匹配测试框架（Jest/Vitest/Mocha 等）、断言风格和目录结构。

**第三步：覆盖边界情况**

```text
为通知服务添加边界条件测试，包括：
- 空输入
- 超长字符串
- 并发发送
- 网络超时
```

**第四步：运行并修复**

```text
运行新添加的测试，修复所有失败。
```

### 提效技巧

| 技巧 | 说明 |
|------|------|
| **说明要验证什么行为** | "测试 token 过期后的刷新逻辑" 比 "加点测试" 好 |
| **指定不要 mock** | 如果你偏好集成测试，明确说 "避免使用 mock" |
| **只跑单个测试文件** | `运行 src/auth/__tests__/login.test.ts` 比 "跑全部测试" 快得多 |
| **让 Claude 发现盲区** | "分析代码路径，建议我可能遗漏的边界情况" |

### 常见陷阱

> **陷阱**：Claude 写了很多测试，但测试内容都是"happy path"，没有真正覆盖到边界。
>
> **应对**：明确列出你关心的边界条件，或者让 Claude 先分析代码路径再设计测试用例。

## 场景五：提交、分支、PR 描述

### 适用时机

- 完成一轮开发后提交代码
- 创建 Pull Request
- 生成变更日志

### 快速提交

```text
用描述性的 commit 信息提交我的更改。
```

Claude 会自动读取 `git diff`，生成有意义的 commit 消息。

### 创建 PR

最简单的方式：

```text
为我的更改创建一个 PR。
```

如果需要更精细的控制：

**第一步：总结变更**

```text
总结我对认证模块的所有更改。
```

**第二步：创建 PR**

```text
创建 PR，标题要清晰，描述要包含：
- 变更动机
- 实现方案
- 测试情况
- 潜在风险
```

**第三步：完善描述**

```text
在 PR 描述中补充安全改进的上下文。
```

### PR 与会话的关联

当你通过 `gh pr create` 创建 PR 后，会话会自动关联到该 PR。之后可以用以下方式恢复：

```bash
claude --from-pr 42
```

### Git 工作流速查

| 操作 | 提示词示例 |
|------|----------|
| 提交 | `提交更改，commit 信息用英文` |
| 创建分支 | `创建 feature/oauth-login 分支并切换过去` |
| 创建 PR | `创建 PR` |
| 查看 diff | `显示我和 main 分支的差异` |
| 交互式 rebase | `把最近 3 个 commit 压缩成一个` |
| Cherry-pick | `从 feature-x 分支 cherry-pick 修复 auth 的那个 commit` |

## 场景六：单会话代码审查

### 适用时机

- 在合并前自己先过一遍代码
- 审查他人的 PR
- 安全审计

### 基本审查

```text
审查我最近的代码更改，关注：
- 潜在的 Bug
- 安全漏洞
- 性能问题
- 代码风格一致性
```

### 针对性审查

```text
检查 @src/api/payment.ts 的安全性，
特别关注注入漏洞和敏感数据处理。
```

### 使用 Subagent 进行隔离审查

更好的做法是使用 subagent 进行审查，因为审查者不应该有编写代码时的偏见：

```text
使用 subagent 审查我刚才的所有更改，
关注边界情况、竞态条件和与现有代码风格的一致性。
```

Subagent 在独立的上下文窗口中运行，审查完成后只返回结论摘要，不会污染你的主会话。

### 常见陷阱

> **陷阱**：在写完代码的同一个会话里让 Claude 审查——它倾向于认为自己写的代码是对的。
>
> **应对**：用 subagent 审查，或者 `/clear` 后再审查，或者新开一个会话专门做 review。

## 组合拳：一轮对话完成"修 Bug → 写测试 → 提交"

前面讲了六个独立场景，但日常开发中最爽的是把它们串起来，一轮对话走完完整闭环。

### 完整示例

**第一步：诊断问题**

```text
用户报告在会话超时后登录失败。
检查 src/auth/ 中的认证流程，特别是 token 刷新逻辑。
先诊断问题，不要改代码。
```

**第二步：编写失败测试**

```text
基于你的诊断，先写一个能重现这个 Bug 的测试用例。
运行测试确认它确实会失败。
```

**第三步：修复并验证**

```text
现在修复这个 Bug。
修完后运行刚才的测试确认通过，
然后运行整个 auth 模块的测试套件确认没有回归。
```

**第四步：提交**

```text
提交所有更改，commit 信息要说明修了什么问题以及如何修复。
```

### 这个流程的关键原则

| 原则 | 说明 |
|------|------|
| **先诊断再动手** | 避免 Claude 直接猜测性修改 |
| **测试先行** | 先写失败测试，确保真的重现了问题 |
| **验证闭环** | 修复后运行测试，不只是"看起来对了" |
| **原子提交** | 一个 commit 解决一个问题 |

## 跨场景的通用最佳实践

### 上下文管理

上下文窗口是你最宝贵的资源。它填满得越快，Claude 的表现越差。

| 策略 | 操作 |
|------|------|
| **任务间清理** | 不相关任务之间用 `/clear` 重置 |
| **限定探索范围** | "只看 `src/auth/`" 而不是 "看整个代码库" |
| **用 subagent 做调查** | 让 subagent 读文件并汇报，避免主会话上下文膨胀 |
| **及时压缩** | 运行 `/compact` 或等待自动压缩触发 |
| **快速问题用 `/btw`** | 答案出现在覆盖层中，不进入对话历史 |

### 验证驱动

Claude Code 最高杠杆的最佳实践：**给 Claude 一种验证自己工作的方式**。

| 验证方式 | 适用场景 |
|----------|----------|
| 运行测试套件 | 所有代码变更 |
| 运行 linter/type check | 风格和类型安全 |
| 运行构建命令 | 确认编译通过 |
| 截图对比 | UI 变更 |
| 自定义脚本 | 特定业务逻辑验证 |

```text
实现 validateEmail 函数。
测试用例：user@example.com 为真，invalid 为假，user@.com 为假。
实现后运行测试确认全部通过。
```

### 纠偏手段

| 手段 | 快捷键/命令 | 效果 |
|------|-----------|------|
| **中断** | `Esc` | 停止当前操作，保留上下文 |
| **回退** | `Esc + Esc` 或 `/rewind` | 回到之前的检查点 |
| **撤销** | 直接说"撤销刚才的更改" | Claude 恢复修改 |
| **重置** | `/clear` | 清空上下文重新开始 |
| **分支** | `/branch` | 从当前点分叉出新的对话分支 |

> **经验法则**：如果你在一个会话中纠正了 Claude 两次以上，上下文已经被失败方案污染。`/clear` 后用更好的提示重新开始，几乎总是比继续纠正更高效。

## 使用 `@` 引用加速工作

`@` 是 Claude Code 中最实用的快捷语法，能让你直接把文件、目录、甚至 MCP 资源引入对话上下文。

| 引用方式 | 示例 | 效果 |
|----------|------|------|
| **单个文件** | `解释 @src/utils/auth.js 的逻辑` | 文件完整内容加入对话 |
| **目录** | `@src/components 的结构是什么？` | 目录列表（不含内容） |
| **多文件** | `对比 @file1.js 和 @file2.js` | 两个文件内容都加入对话 |
| **MCP 资源** | `查看 @github:repos/owner/repo/issues` | 从 MCP 服务器获取数据 |

> **注意**：`@` 引用文件时，Claude 也会自动加载该文件所在目录和父目录的 `CLAUDE.md`，这是有意为之——确保 Claude 理解目录级别的规则。


```text

```

## 让 Claude 作为 Unix 工具

Claude Code 不仅是交互式工具，还可以嵌入到脚本和管道中。

### 管道输入输出

```bash
# 分析构建错误
cat build-error.txt | claude -p '简洁解释这个构建错误的根因' > analysis.txt

# 批量检查
find src -name "*.ts" -newer last-check | \
  xargs -I {} claude -p "检查 {} 中的 TypeScript 类型错误"
```

### 作为 Linter 使用

```json
// package.json
{
  "scripts": {
    "lint:claude": "claude -p 'you are a linter. look at changes vs. main and report typos. report filename and line number on one line, description on the second line. no other text.'"
  }
}
```

### 输出格式控制

| 格式 | 标志 | 适用场景 |
|------|------|----------|
| **纯文本** | `--output-format text` | 默认，简单集成 |
| **JSON** | `--output-format json` | 需要完整对话日志 |
| **流式 JSON** | `--output-format stream-json` | 实时处理每轮输出 |

## 并行会话与 Worktrees

当你需要同时处理多个任务时，Claude Code 支持用 Git Worktrees 隔离每个会话的代码环境。

### 快速创建

```bash
# 在隔离的 worktree 中启动 Claude
claude --worktree feature-auth

# 另一个终端，另一个任务
claude --worktree bugfix-123

# 自动生成名称
claude --worktree
```

### Worktree 清理

| 情况 | 处理 |
|------|------|
| 会话结束时无更改 | 自动删除 worktree 和分支 |
| 有未提交的更改 | Claude 提示你选择保留或删除 |

### 复制环境文件

新 worktree 不包含 `.env` 等 gitignored 文件。在项目根目录添加 `.worktreeinclude`：

```text
.env
.env.local
config/secrets.json
```

> **提示**：将 `.claude/worktrees/` 加入 `.gitignore`，避免 worktree 内容在主仓库中显示为未跟踪文件。

## 定时任务与自动化

Claude Code 支持按计划自动运行任务。

| 调度方式 | 运行位置 | 最适合 |
|----------|----------|--------|
| **云计划任务** | Anthropic 基础设施 | 电脑关机也能运行的任务 |
| **Desktop 计划任务** | 本地机器 | 需要访问本地文件的任务 |
| **GitHub Actions** | CI 管道 | 与仓库事件相关的任务 |
| **`/loop`** | 当前 CLI 会话 | 会话打开时的快速轮询 |

```text
为计划任务编写提示时，明确说明成功标准和结果处理方式。
任务自主运行，不能提出澄清问题。

例如："审查标记为 needs-review 的 PR，对问题留下评论，
并在 #eng-reviews 频道发布摘要。"
```

## 恢复与切换会话

开发不是线性的——你可能在修 Bug 到一半时被叫去看另一个问题。Claude Code 的会话管理支持这种工作方式。

### 恢复会话

```bash
# 继续最近的对话
claude --continue

# 从列表中选择
claude --resume

# 按名字恢复
claude --resume auth-refactor

# 恢复关联到某个 PR 的会话
claude --from-pr 42
```

### 命名会话

```bash
# 启动时命名
claude -n auth-refactor

# 会话中命名
/rename auth-refactor
```

### 会话选择器快捷键

| 快捷键 | 操作 |
|--------|------|
| `↑` / `↓` | 导航会话列表 |
| `Enter` | 恢复选中会话 |
| `P` | 预览会话内容 |
| `R` | 重命名会话 |
| `/` | 搜索过滤 |
| `A` | 切换当前目录/所有项目 |
| `B` | 过滤当前 Git 分支的会话 |

> **最佳实践**：像对待 Git 分支一样对待会话——不同的工作流用不同的会话，及时命名，方便回溯。

## 常见失败模式与应对

| 失败模式 | 症状 | 修复方法 |
|----------|------|----------|
| **厨房水槽会话** | 一个会话里混杂多个不相关任务 | 不同任务之间用 `/clear` |
| **反复纠正** | 纠正了两次 Claude 还做不对 | `/clear` 后用更好的提示重新开始 |
| **无限探索** | Claude 读了上百个文件 | 限定范围或用 subagent |
| **跳过验证** | 代码看起来对但实际有问题 | 始终提供验证标准 |
| **过度依赖单会话** | 上下文窗口满了导致质量下降 | 拆分任务到多个会话 |

## 本篇小结

| 场景 | 核心提示模式 | 验证方式 |
|------|------------|----------|
| **理解代码** | "给我概览 → 深入模块 → 追踪调用链" | 自己确认理解正确 |
| **修 Bug** | "分享错误 → 诊断 → 修复 → 运行测试" | 测试通过 |
| **重构** | "Plan Mode 探索 → 规划 → 实现 → 验证" | 测试 + 构建通过 |
| **写测试** | "找未覆盖代码 → 生成测试 → 补边界 → 运行" | 测试通过 + 覆盖率 |
| **Git/PR** | "提交 → 创建 PR → 完善描述" | PR 审查 |
| **代码审查** | "用 subagent 审查变更" | 独立视角 |

**最重要的三条原则**：

1. **给 Claude 验证方式**——测试、lint、构建命令，让它能自查
2. **管理上下文**——`/clear`、subagent、`/compact`，保持上下文窗口干净
3. **及时纠偏**——`Esc` 中断、`/rewind` 回退、两次纠正后重新开始
