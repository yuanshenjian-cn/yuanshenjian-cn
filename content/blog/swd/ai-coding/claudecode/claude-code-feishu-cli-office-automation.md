## 一、为什么飞书 CLI 值得单开一篇

Claude Code 自身只管两件事：读取上下文、执行命令。它要能"操作飞书"，必须有一层稳定的执行入口。

在过去，这一层只有两个选择：

- 自己调飞书开放平台的 REST API，处理鉴权、分页、错误重试，写很多胶水代码；
- 走 MCP 服务，能力依赖第三方实现，常常落后于官方接口。

`@larksuite/cli`（lark-cli）改变了局面。这是飞书官方维护、面向 AI Agent 设计的命令行工具，自 2026 年开源以来快速迭代，**截至 2026 年 5 月**已经达到这样的规模：

| 维度 | 当前数据 |
|------|---------|
| 业务域覆盖 | **17 个**（消息、文档、Base、Sheets、Slides、日历、邮箱、任务、会议、知识库、通讯录、Drive、考勤、审批、OKR、画板、Markdown） |
| 精选命令 | **200+** 条 |
| AI Agent Skill | **25 个**（实际安装后清点；README 仍写 24，差一个新增的 `lark-vc-agent`） |
| Raw API 覆盖 | **2500+** 个开放平台端点 |
| GitHub Stars | 9.9k+，仓库地址 [`larksuite/cli`](https://github.com/larksuite/cli) |
| 许可证 | MIT |

对 Claude Code 来说，这意味着：**只要把 lark-cli 装好、登录通过，就直接拥有了一组可以拼接组合的"操作飞书"的工具**，连 MCP 配置都可以省。

## 二、安装与登录：四条命令把基座搭好

官方安装指南位于 [open.feishu.cn 的 mcp-archive 目录](https://open.feishu.cn/document/no_class/mcp-archive/feishu-cli-installation-guide.md)。下面是核实过的、当前可用的最小流程。

### 1. 环境检查

```bash
node --version   # 任意 LTS 即可
npm --version
```

> 仅当你想从源码构建时，才需要 Go 1.23+ 与 Python 3；通过 npm 全局安装的话上面两条就够了。

### 2. 安装 CLI 与必需的 Skill

```bash
# 全局安装 CLI 主体
npm install -g @larksuite/cli

# 安装配套的 Agent Skills（实际安装后是 25 个，必装）
npx skills add larksuite/cli -y -g
```

`skills add` 这一步常被忽略。CLI 本体只提供命令；Skills 才是 Claude Code 在对话里调度命令、组装参数、串联多步操作的"提示与编排"层，二者缺一不可。

### 3. 应用凭证初始化

```bash
lark-cli config init
```

执行后 CLI 会给出一个授权链接。如果是在 Claude Code 里跑，链接会被打印在终端，**Agent 应当把链接原样递给你，由你在浏览器里完成授权**，不要让 Agent 自动点开。

### 4. 登录与状态校验

```bash
# 推荐参数：自动勾选常用 scope
lark-cli auth login --recommend

# 校验登录状态
lark-cli auth status
```

成功后凭证会保存在操作系统的原生 keychain 里，CLI 本身不写明文配置。

### 5. 把整套指引交给 Agent

懒得人工跑命令时，可以把下面这段贴进 Claude Code 让它代办，遇到需要授权的环节再交回给你：

```text
帮我安装并初始化飞书 CLI。安装指引在：
https://open.feishu.cn/document/no_class/mcp-archive/feishu-cli-installation-guide.md
要求：
1. 用 npm 全局安装，同时执行 skills add；
2. 跑 config init 与 auth login --recommend；
3. 遇到授权链接时，先暂停并把链接原样交给我，不要自行打开浏览器。
```

## 三、套路一：把"重复会"沉淀成跨场次知识库

### 适用场景

公司里那种**每周固定、议题相似**的会议：选题会、周会、复盘会、培训会、产品例会。妙记和文档单独看都还行，连起来回看就丢了。

### 思路

让 Claude Code 通过 lark-cli 抓取一段时间内的同类会议妙记，按"会议清单 → 议题 → 决议 → 原因 → 后续动作"五段式整理，再写回飞书云文档形成可追加的知识库。

### 一条提示词跑通

```text
请用 lark-cli：
1. 在妙记里检索 2026-03 月所有名称包含「选题会」的记录；
2. 对每场会议提取：日期、与会人、讨论选题、最终决议、决议理由、需跟进事项；
3. 汇总成飞书云文档「选题会知识库」，按"会议清单 / 选题转化 / 节奏分析 / 决策模式 / 待跟进项"五个一级标题分块；
4. 在文档开头追加一条「下次更新策略」：每周一自动抓上一周的选题会。
```

### 经验值得固化的几点

- **不要让 Agent 一次抓三个月以上的妙记**，调用次数和 token 都会爆。按月或按周分批写入，更稳。
- **决议理由比决议本身更值钱**。新人最缺的就是"为什么当初这么定"的语境，提示词里务必显式要求。
- **存到云文档而不是本地**：知识库的复用价值在协同，不在归档。

## 四、套路二：跨数据源的工作复盘

### 适用场景

需要做季度汇报、晋升答辩、年终总结时，材料散落在私信、群聊、妙记、日程、任务、邮件、文档、OKR 里，人工拼接极费时间。

### 思路

让 Agent **并行拉取**多个 Skill 的数据，再合成一份「时间分布 + 协作图谱 + 主线项目 + 产出物 + 改进建议」结构的文档。

### 核心提示词模板

```text
请帮我做一份过去 90 天的工作复盘：
- lark-im：导出我参与的群聊话题分布与高频协作人；
- lark-calendar：统计会议总时长、最常出现的会议系列、空闲时段比例；
- lark-vc + lark-minutes：抽取我作为主讲或决策方的会议纪要要点；
- lark-task：列出我创建/完成/逾期的任务，按项目聚合；
- lark-mail：识别外发邮件的主要主题；
- lark-doc + lark-drive：盘点这段时间产出的文档与文件；
- lark-okr：拉取我的 OKR 当前进度与对齐关系。

最后生成一份飞书文档，包含：
1. 一句话画像；
2. 时间投入分布饼图（用表格或 Markdown 表达即可）；
3. 主要协作对象 Top 10；
4. 三个主线项目及代表性产出；
5. 三条值得改进的洞察（不要泛泛而谈，要有数据支撑）。
```

### 可复用做法

- **把这个流程固化为 Skill**。Claude Code 支持自定义 Skill，把上面这段提示词、并行抓取策略、输出模板抽出来，团队所有人一句话就能跑。
- **季度报告之外**，同一套骨架可以做：新员工入职 30 天总结、项目结项报告、跨部门协作健康度评估。
- **数据隐私先想清楚**：复盘内容默认只给本人看，文档权限不要设成「组织内可见」。

## 五、套路三：群聊机器人接管重复对账

### 适用场景

财务对账、回款核对、合同续签确认、订单异常追踪——任何"查表 → 群里 @ 对方确认 → 根据反馈分流给不同同事"的循环操作。

### 流程拆解

```
机器人被 @ 后开始执行：
  ↓
[lark-base]  从多维表格按条件检索数据
  ↓
[lark-im]    在群里 @ 目标对象，贴出明细，请求确认
  ↓
[lark-event] 监听对方回复事件（WebSocket）
  ↓
分流决策：
  · 确认无误 → @ 发起人，标记完成
  · 金额不符 → 私信商务负责人
  · 漏单     → 私信财务负责人
  · 其他     → 转人工
```

### 关键能力点

- `lark-event` 提供事件订阅，正则路由把"博主回复"这类异步消息变成可编排的步骤；
- `lark-im` 在 2026 年新增了 **以本人身份发消息** 的能力（默认仍是机器人身份，按需切换 `--as user`），私聊场景的触达率因此更高；
- 多维表格的字段、视图、权限都可以由 `lark-base` 维护，对账明细模板不必硬编码。

### 一段把需求"口喷"过去也能跑通的提示词

```text
帮我做一个飞书机器人，逻辑如下：
1. 把博主和我拉进同一个群后，群里有人 @ 机器人「对账」，机器人就去多维表格
   「博主合作明细」检索该群成员对应的本月项目，输出：项目名、金额、备注；
2. 机器人在群里 @ 博主请求确认；
3. 监听博主回复：
   - 「无误」或「OK」→ @ 我并标记完成；
   - 含「金额」→ 把原消息转发到群「商务核对」并 @ 张三；
   - 含「漏单」→ 把原消息转发到群「财务补单」并 @ 李四；
   - 其它内容 → 私信我请求介入。
4. 整个过程的关键节点都写入「对账日志」表格，留痕。
```

### 风险点

- 群聊机器人能力强，**误操作的代价也大**。先用真实的小流程试跑，确认决策树和兜底分支都到位再放量；
- 涉及金额、合同的确认环节，**永远要给「人工接管」一个出口**，不要让 Agent 在边界条件下自作主张；
- 群里贴明细前先脱敏，敏感字段（如完整身份证号、银行卡号）走私信。

## 六、套路四：从一句话到协同画板

### 适用场景

开会讨论架构图、流程图、用户旅程图，过去要么手绘要么会后专人补图。现在可以让 Agent 当场出，而且**生成的是飞书原生画板，所有人能直接拖拽、改字、加节点**。

### 与传统"AI 生成 HTML 信息图"的差异

| 维度 | 自生成 HTML | 飞书画板（lark-whiteboard） |
|------|------------|----------------------------|
| 协同编辑 | 难，需重新生成 | 原生支持，任何人打开即可改 |
| 评论留痕 | 没有 | 节点级评论 |
| 与文档联动 | 链接外挂 | 可嵌入文档/Wiki/Base |
| 后续维护 | 越改越乱 | 同一份画板持续演进 |

### 示例提示词

```text
请基于下面这段口述，生成一张飞书画板：

「用户从扫码入场，先到欢迎屏看议程；30 分钟前可以进主会场；
 茶歇区有签到二维码，扫码后会自动加入对应群；
 主论坛结束后分流到三个分论坛；分论坛结束统一发反馈问卷。」

要求：
- 用泳道图（横向流程）表达；
- 节点之间的判断分支显式画出；
- 每个分论坛各占一列；
- 把画板共享到群「AIFUT 筹备」。
```

### 不只是流程图

`lark-whiteboard` Skill 渲染的 DSL 同样可以产出脑图、组织架构图、思维导图，甚至简单 PPT。这一点和 HTML 信息图的最大区别在于：**画板是一种活的文档**，可以一直长在团队里。

## 七、套路五：报销与审批的全流程托管

### 适用场景

报销、出差申请、采购申请、合同审批——所有"按 SOP 收集材料 → 套模板 → 提交 → 跟踪反馈"的流程。

### 端到端流程

```
1. 提报方：
   Claude Code 读取报销 SOP（存放在 lark-wiki）
   → 调 lark-mail 检索「fapiao@」开头的邮件，下载 PDF 附件
   → 解析发票项、金额、抬头
   → 调 lark-approval 准备表单字段
   → 提交前给本人确认，确认后正式发起

2. 审批方：
   Claude Code 调 lark-approval 拉取待办
   → 调 lark-base 核对预算、调 lark-doc 核对项目背景
   → 把结论与依据汇总后给本人确认
   → 确认后批准/驳回，并附带审批意见
```

### 关键 Skill

- `lark-approval`：审批任务的增删改查、批准/拒绝/转交/撤销/抄送；
- `lark-mail`：发票邮件检索与附件下载；
- `lark-wiki`：调取报销 SOP；
- `lark-shared`：身份切换（提报用本人，机器人审批要明示）。

### 让流程稳的几条规则

- **任何"动钱"的步骤都必须人工确认**。把"提交前 confirm"和"批准前 confirm"写进 Skill 模板，关掉就别用。
- **审批意见要写依据而非结论**。提示词里要求 Agent 给出"依据 X 文档 / Y 表格的 Z 字段"这样的引用，方便事后审计。
- **SOP 放 Wiki 而非个人文档**，便于版本管理与权限收敛。

## 八、把五个套路抽象成一个通用骨架

回看这五个场景，能提炼出一个统一的执行模板：

```
[识别数据源] → [选择对应 Skill] → [按字段抓取] → [合成文档/画板/消息] → [人工确认或自动分流]
```

| 步骤 | 对应能力 | 高频陷阱 |
|------|---------|---------|
| 识别数据源 | `lark-shared` 列出可用 Skill | 数据散在三个以上系统时容易遗漏 |
| 抓取数据 | `lark-base` / `lark-im` / `lark-vc` 等 | 时间范围过大、未分页 |
| 合成产出 | `lark-doc` / `lark-whiteboard` / `lark-sheets` | 一次性生成体量过大，容易丢内容 |
| 人工确认 | `lark-shared` + 自定义 Skill | 不留确认环节，事故责任不清 |
| 分流通知 | `lark-im` / `lark-mail` / `lark-event` | 通知频率失控，打扰团队 |

写自己的 Skill 时按这个骨架填空，再用 `--dry-run` 在不真正修改飞书数据的前提下预演一遍，是最稳妥的做法。

## 九、边界与风险：用之前先想清楚

飞书 CLI 在 README 里自己写得很直白，照搬过来作为提醒：

- **不要把"以本人身份发消息"的能力轻易接入群聊机器人**。一次提示注入就可能借你的账号广播错误信息；
- **AI 操作仍会幻觉**。涉及付款、合同、对外公告的环节，必须有人类二次确认；
- **审计留痕优于事后追溯**。所有关键动作（审批、转账、对账完成）都让 Agent 写一条飞书消息或表格记录；
- **权限按需收敛**。`auth login --recommend` 选的是常用 scope，但生产用途建议改为按 Skill 显式申请；
- **凭证写在 keychain 里**，不要 export 成环境变量到处传；
- **官方风险声明**：CLI 建议作为私人对话助手使用，谨慎拉入多人群聊。

---

写到这里其实可以下个判断：**Agent 真正落地公司，"动数据的接口"比"会写代码"更关键**。当组织里 80% 的协同事实都沉在某一个平台上时，给 Agent 一根接进这个平台的电源线，比让它在本地多读几个仓库要划算得多。飞书 CLI 现在的覆盖密度已经过了一个临界点，剩下的问题，是团队愿不愿意把流程改造到能让 Agent 接管。
