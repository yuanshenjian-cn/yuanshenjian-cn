仓库现在叫 `oh-my-openagent`，推荐的 plugin entry 也是 `oh-my-openagent`，配置文件我也建议直接用 `oh-my-openagent.json[c]`；但 npm 包名和命令行仍然保留 `oh-my-opencode`，旧的 `oh-my-opencode.json[c]` 也还兼容。下面提到安装命令时我沿用旧命令名，提到仓库与配置时则优先用新名字。

我把这些 Agent 分成两大类来看：Primary Agents（主 agent）和 Sub Agents（子 agent）。

**图例说明**:

- **Primary**: 主 agent，可直接作为入口调用
- **Sub agent**: 子 agent，通常由主 agent 自动调度
- **EXP**: 昂贵（使用高级模型）
- **CHEAP**: 便宜（性价比高）
- **FAST**: 快速、轻量、适合搜索类任务

### Agent 分类

```
Primary Agents（主 agent）: 4个 - 负责任务编排和执行
├── Sisyphus（西西弗斯）- 主编排
├── Hephaestus（赫菲斯托斯）- 深度实现
├── Atlas（阿特拉斯）- 会话编排
└── Prometheus（普罗米修斯）- 战略规划

Sub Agents（子 agent/专家）: 7个 - 提供专业领域支持
├── Metis（墨提斯）- 预规划分析
├── Momus（摩摩斯）- 计划验证
├── Oracle（神谕）- 设计/调试顾问
├── Librarian（莱布瑞恩）- 文档/代码搜索
├── Explore（艾克斯）- 快速代码探索
├── Multimodal Looker（莫达尔）- 图像/PDF分析
└── Sisyphus-Junior（小西西弗斯）- 子任务执行
```
> 💡 **Agent 分类说明**：我更喜欢按实际工作流来记。Sisyphus 和 Hephaestus 属于主执行链路，Prometheus 负责 interview mode 规划，Oracle / Librarian / Explore / Multimodal Looker / Metis / Momus 则是被主代理拉起来的专业助手。


### 核心设计哲学

- **专业化分工**: 每个 agent 都有明确的职责边界
- **协作而非竞争**: agent 之间相互配合，形成完整工作流
- **推石头精神**: 任务不完成绝不停止（Sisyphus 核心机制）

## Primary Agents（主 agent）

主 agent 是你日常使用的主力，可以直接调用或作为入口点，使用`Tab`可以快速切换。

### Sisyphus | 西西弗斯

| 属性 | 详情 |
|------|------|
| **角色** | 推石头者 |
| **中文名** | 西西弗斯 |
| **希腊神话原型** | 被惩罚永远推石头上山的智者 |
| **模型** | `anthropic/claude-opus-4-7` |
| **Fallback** | `kimi-k2.5` → `glm-5` |
| **温度** | 0.1 |
| **成本** | EXPENSIVE |

**核心定位**: 主编排器、团队协调者

**工作机制**:

```
用户请求 → 西西弗斯分析 → 并行启动 sub agent:
  ├── 艾克斯: 快速找代码
  ├── 莱布瑞恩: 查官方文档
  ├── 神谕: 给架构建议
  └── 赫菲斯托斯: 执行具体实现
  ↓
汇总结果 → 持续推动直到100%完成
```

**特点**:

- 🔱 不自己埋头苦干，而是组建团队并行工作
- ⚡ 核心机制是"推石头"——任务不完成绝不停止
- 🎯 擅长复杂项目管理、多 agent 协调

**读写权限**: ✅ 全权限（可以创建任务、调用其他 agent、读写文件）

**使用场景**:

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
重构整个认证系统

# 使用 ulw 魔法词（最推荐）
ulw 优化这个React应用的性能
```

### Hephaestus | 赫菲斯托斯

| 属性 | 详情 |
|------|------|
| **角色** | 锻造之神 |
| **中文名** | 赫菲斯托斯 |
| **希腊神话原型** | 火与锻造之神，众神的工匠 |
| **模型** | `openai/gpt-5.4` |
| **Fallback** | 按当前 provider 可用性自动回退 |
| **温度** | 0.1 |
| **成本** | EXPENSIVE |

**核心定位**: 自主深度工作者、端到端实现者

> 💡 **备选思路**：如果 `gpt-5.4` 路由不可用，我会优先看当前可用的长上下文或高执行力模型，而不是把某个备选型号写死在配置里。

**工作机制**:

```
用户目标 → 赫菲斯托斯启动预研:
  ├── 启动 2-5 个 艾克斯 agent 并行探索
  ├── 启动 莱布瑞恩 研究最佳实践
  └── 分析项目风格模式
  ↓
制定计划 → 独立执行 → 验证测试 → 完成交付
```

**特点**:

- 🔥 "正当工匠"——不抄近路，methodical 执行
- 🎨 先探索研究，再精准实现
- 🎯 给他一个目标，他自己搞定一切

**读写权限**: ✅ 全权限（但不能创建任务，只能执行）

**使用场景**:

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
给我实现一个完整的用户认证系统

# 深度重构
将这个单体应用拆分为微服务
```

**与西西弗斯的区别**:

- **西西弗斯**: 我给你规划，让团队来做
- **赫菲斯托斯**: 我自己研究，我自己做，直到完成

| 属性 | 详情 |
|------|------|
| **角色** | 擎天神 |
| **中文名** | 阿特拉斯 |
| **希腊神话原型** | 撑起天空的泰坦 |
| **模型** | 以当前 features reference 为准 |
| **Fallback** | 以当前工作流配置为准 |
| **温度** | 0.1 |
| **成本** | EXPENSIVE |

**核心定位**: 会话级编排者、大型项目协调者

**工作机制**:

```
大型项目启动 → 阿特拉斯接管:
  ├── 分析整体架构
  ├── 规划阶段性目标
  ├── 协调 西西弗斯 执行具体阶段
```

- 🏗️ 在更大时间尺度上协调工作
- ⚖️ 适合长期项目（数小时到数天）

**读写权限**: ✅ 以当前工作流配置为准，核心职责仍是编排和验证

Atlas 这条线我更把它看成特定工作流里的会话级编排者，而不是日常默认入口。

**使用场景**:

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
```

**与西西弗斯的区别**:

- **西西弗斯**: 任务级编排（单个任务内）
- **阿特拉斯**: 会话级编排（跨多个任务/会话）

### Prometheus | 普罗米修斯

| 属性 | 详情 |
|------|------|
| **角色** | 先知 |
| **中文名** | 普罗米修斯 |
| **希腊神话原型** | 为人类盗取火种的泰坦 |
| **模型** | `anthropic/claude-opus-4-7` / `kimi-k2.5` / `glm-5` |
| **Fallback** | 与 Sisyphus 同路由 |
| **温度** | 0.1 |
| **成本** | EXPENSIVE |

**核心定位**: 战略规划者、计划制定者

**⚠️ 重要限制: 只计划，不写代码！**

**工作机制**:

```
用户需求 → 普罗米修斯深度分析:
  ├── 墨提斯: 预规划咨询
  ├── 莱布瑞恩: 调研技术方案
  └── 摩摩斯: 验证可行性
  ↓
生成详细工作计划 → 交给 赫菲斯托斯/西西弗斯 执行
```

**特点**:

- 🔮 深度分析，制定完整计划
- 📋 从不实现，专注规划
- 🎯 项目启动前的战略规划

**读写权限**: ❌ **只读**（禁止 write, edit, task, call_omo_agent）

**使用场景**:

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
帮我规划这个系统的架构

# 分析实现方案
分析实现这个功能的最佳方案
```

**当普罗米修斯完成规划后，我通常会直接继续执行 `/start-work` 这条链路。**

这个动作本质上是调用 Prometheus 的 interview mode 规划结果，继续进入后续执行阶段，而不是 OpenCode core 自带的模式切换开关。

**执行流程：**

```
用户确认计划 → /start-work
    ↓
切换到 Atlas（擎天神）执行模式
    ↓
Atlas 接管会话级编排:
  ├── 分析整体架构
  ├── 规划阶段性目标
  ├── 协调 西西弗斯 执行具体阶段
  └── 监控进度，处理阻塞
    ↓
西西弗斯 并行启动 sub agent:
  ├── 艾克斯: 快速找代码
  ├── 莱布瑞恩: 查官方文档
  ├── 神谕: 给架构建议
  └── 赫菲斯托斯: 执行具体实现
    ↓
持续推动直到 100% 完成
```

**权限变化**: 从只读规划链路进入后续执行链路。

## Agent 协作机制

### 协作关系图

```
用户请求
    │
    ▼
┌─────────────────────────────────────┐
│         Primary Agents              │
```

### 调用链示例

**典型工作流**:

```
用户: "实现用户认证"
    ↓
普罗米修斯: 制定计划
    ├── 墨提斯: 需求分析
    ├── 莱布瑞恩: 调研JWT
    └── 摩摩斯: 验证方案
    ↓
西西弗斯: 协调执行
    ├── 艾克斯: 找现有代码
    ├── 神谕: 安全建议
    └── 赫菲斯托斯: 实现代码
    ↓
任务完成
```

## 使用场景与示例

### 场景 1: 快速开发功能（最常用）

**需求**: 实现一个完整的用户认证系统

**推荐入口**: Hephaestus（赫菲斯托斯）

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
给我实现一个完整的用户认证系统，包括注册、登录、密码重置
```

**执行过程**:

1. 赫菲斯托斯启动预研，调用多个艾克斯并行搜索
2. 莱布瑞恩研究 JWT 最佳实践
3. 神谕提供安全建议
4. 赫菲斯托斯独立完成实现
5. 自动测试验证

**预计耗时**: 10-30分钟（取决于复杂度）

### 场景 2: 复杂项目重构

**需求**: 重构整个前端架构

**推荐入口**: Sisyphus（西西弗斯）+ `ulw` 关键词

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
ulw 重构整个前端架构，从 Class 组件迁移到 Hooks
```

**执行过程**:

1. 西西弗斯分析整体影响
2. 协调多个 agent 并行工作
3. 小西西弗斯批量处理不同模块
4. 持续推动直到全部完成

**预计耗时**: 1-数小时（ultrawork 模式会持续执行）

### 场景 3: 先规划再执行

**需求**: 开发一个新的电商平台

**推荐入口**: Prometheus（普罗米修斯）→ Hephaestus（赫菲斯托斯）

**步骤 1**: 规划阶段

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
我要做一个电商网站，帮我规划一下架构和技术选型
# 会自动触发普罗米修斯
```

**步骤 2**: 执行阶段（根据生成的计划）

```bash
# 在 TUI 中继续输入：
按照刚才的计划，开始实现用户模块
# 会自动触发赫菲斯托斯
```

### 场景 4: 快速定位代码

**需求**: 找到特定功能的实现位置

**推荐入口**: Explore（艾克斯）

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
找到认证中间件的实现代码
# 会自动触发艾克斯（最快最经济）
```

**预计耗时**: 几秒钟

### 场景 5: Debug 问题

**需求**: API 响应很慢，需要诊断

**推荐入口**: Oracle（神谕）

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
这个API响应很慢，帮我找出原因并优化
# 会自动触发神谕，然后由赫菲斯托斯实施修复
```

**执行过程**:

1. 神谕诊断问题根因
2. 艾克斯搜索相关代码
3. 莱布瑞恩查性能优化文档
4. 赫菲斯托斯实施修复

### 场景 6: 研究技术方案

**需求**: 了解新技术的工作原理

**推荐入口**: Librarian（莱布瑞恩）

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
React Server Components 的工作原理是什么？
# 会自动触发莱布瑞恩
```

**预计耗时**: 1-2分钟


## 最佳实践

### 懒人模式（推荐）

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
ulw 你的需求描述
# 一切交给西西弗斯
```

### 精准控制

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
用赫菲斯托斯实现用户注册功能

# 或先规划再执行
用普罗米修斯规划这个系统
```

### 分阶段工作

```bash
# 终端执行
opencode

# 阶段1: 规划（在 TUI 中输入）
帮我规划电商网站架构

# 阶段2: 实现（继续在 TUI 中输入）
按照刚才的计划，开始实现用户模块

# 阶段3: 优化（继续在 TUI 中输入）
优化刚才实现的代码性能
```

### 充分利用 sub agent

```bash
# 终端执行
opencode

# 进入 TUI 后输入：
莱布瑞恩，查一下 React 19 的新特性

# 快速定位代码
艾克斯，找到所有使用 Redux 的地方
```

| Multimodal Looker | `gpt-5.4` | 跟随当前多模态链路 |
| Metis | 与规划链路同级别模型 | 跟随当前规划链路 |
